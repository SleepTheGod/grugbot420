0. WHAT THIS IS

this document describes a system of symbolic computation where typed handles called sigils serve as structural priors for input processing, pattern matching, and automated reasoning. sigils are not passive tokens. they are active macro-expansion sites that rewrite input into executable automaton instructions. the result is a computational model where training is definition, inference is execution, and context is state.

the core implementation lives in three files of the grugbot420 codebase: 

  src/SigilRegistry.jl    -- the symbol table (1168 lines)
  src/SigilPromoter.jl    -- the input rewriter (635 lines)
  src/ArithmeticEngine.jl -- a concrete automaton (439 lines)

everything below derives from what those files actually do, plus architectural extensions that follow from the same principles.

1. SIGIL REGISTRY

1.1 the sigil entry

every sigil is a typed symbolic handle registered in a central SigilRegistry. each entry carries enough metadata to drive promotion, matching, expansion, and voting without ad-hoc logic. the struct: 

struct SigilEntry
    name              :: Symbol    # e.g. :n, :word, :temporal
    class             :: Symbol    # one of SIGIL_CLASSES
    applies_at        :: Symbol    # one of SIGIL_APPLIES_AT
    sigil_type        :: Symbol    # e.g. :numeric, :alphanumeric
    lexicon           :: Vector{String}  # surface forms this sigil captures
    params            :: Dict{Symbol,Any} # class-specific parameters
    expansion         :: Vector{String}   # macro expansion targets
    provenance        :: Symbol    # :builtin or :user
    promote_at_tokenize :: Bool   # auto-promote during input rewrite?
    promote_predicate   :: Function # conditional promotion gate
end

the registry itself is a Dict{Symbol, SigilEntry} keyed by name. lookup is O(1). conflict resolution on merge supports three policies: :error (throw), :overwrite (new wins), :keep (old wins).

1.2 sigil classes

seven classes partition the sigil namespace by structural role: 

SIGIL_CLASSES = (:lambda, :macro, :tag, :glue, :functor, :procedure, :relation)
classroleexample
:lambdaparametric position capture&n captures any number
:macroalternation family&word captures any word
:tagannotation markermetadata labels
:gluereserved (join logic)future use
:functorreserved (higher-order)future use
:procedureordered chain expansionPEMDAS-style macros
:relationdynamic relational triples&temporal, &causal

1.3 applies_at domains

each sigil declares where in the processing pipeline it activates: 

SIGIL_APPLIES_AT = (:bind, :match, :vote_shape, :tone, :render,
                    :thesaurus, :drop_table, :inhibit, :relation)
domainmeaning
:bindcaptures a value during input promotion
:matchparticipates in pattern matching
:vote_shapeshapes the voting architecture of nodes
:toneaffects response tone/style
:renderdrives output rendering
:thesaurusprovides synonym expansion
:drop_tablepruning logic for cascade voting
:inhibitsuppresses certain pathways
:relationrelational triple construction

this separation means the same symbol can participate in multiple pipeline stages without collision. a &n that binds a numeric value at :bind time is the same entry that shapes the vote at :vote_shape time.

1.4 default registry

the shipped registry defines these sigils out of the box: 

&n          -- class :lambda, applies_at :bind, captures numeric tokens
&word       -- class :macro,  applies_at :bind, captures any single word
&rest       -- class :macro,  applies_at :bind, captures remaining input
&noun       -- class :macro,  applies_at :bind, captures noun-class words
&op         -- class :lambda, applies_at :bind, captures operator tokens
&temporal   -- class :relation, expands to ["before","after","during",...]
&causal     -- class :relation, expands to ["causes","results in",...]
&spatial    -- class :relation, expands to ["above","below","beside",...]
&possessive -- class :relation, expands to ["has","owns","contains",...]
&similarity -- class :relation, expands to ["resembles","mirrors",...]

users add more. the registry grows. no recompilation needed.

1.5 registration

register_sigil! validates the entry against class constraints, checks for name collisions, and inserts. the validation rules per class: 

# lambda: must have params[:position] (1-based integer)
# macro:  must have non-empty lexicon
# tag:    must have non-empty lexicon
# procedure: must have non-empty expansion, each element is a valid
#            sigil name or literal string
# relation:  must have non-empty expansion, each element is a verb string

pseudocode for the registration gate: 

function register_sigil!(registry, entry)
    # 1. class must be in SIGIL_CLASSES
    assert entry.class in SIGIL_CLASSES

    # 2. applies_at must be in SIGIL_APPLIES_AT
    assert entry.applies_at in SIGIL_APPLIES_AT

    # 3. class-specific validation
    switch entry.class
        case :lambda
            assert haskey(entry.params, :position)
            assert entry.params[:position] > 0
        case :macro
            assert length(entry.lexicon) > 0
        case :procedure
            assert length(entry.expansion) > 0
            for target in entry.expansion
                assert is_valid_sigil_ref(target) or is_literal(target)
        case :relation
            assert length(entry.expansion) > 0
            for verb in entry.expansion
                assert typeof(verb) == String

    # 4. name collision check
    if haskey(registry, entry.name)
        error("sigil already registered: " + string(entry.name))

    # 5. insert
    registry[entry.name] = entry

1.6 pattern resolution

resolve_sigils_in_pattern parses a pattern string containing sigil references (like "&n &op &n") into a structured list of (literal | sigil_ref) segments. this is the bridge between human-readable pattern syntax and the matching engine: 

function resolve_sigils_in_pattern(pattern, registry)
    segments = []
    i = 1
    while i <= length(pattern)
        if pattern[i] == '&'
            # scan to end of sigil name (alphanumeric + _)
            j = i + 1
            while j <= length(pattern) && is_sigil_char(pattern[j])
                j += 1
            name = Symbol(pattern[i+1 : j-1])
            assert haskey(registry, name)
            push!(segments, (:sigil, name))
            i = j
        else
            # literal character
            push!(segments, (:literal, pattern[i]))
            i += 1
    return segments

the result feeds directly into the cascade matching logic. a node's pattern "what is &n &op &n" resolves to: 

  [:literal "what is "] [:sigil :n] [:literal " "] [:sigil :op] [:literal " "] [:sigil :n]

matching then walks the input token stream against this resolved pattern, binding captured values at each sigil position.

2. SIGIL PROMOTION

2.1 two-layer rewriting

the SigilPromoter is the front door of the system. raw user input enters. promoted input exits. the rewriting has two layers: 

  Layer 1: CANONICALIZE
    "two"        -> "2"
    "plus"       -> "+"
    "times"      -> "*"
    "divided by" -> "/"

  Layer 2: SHAPE-PROMOTE
    "2"   -> "&n"   (numeric shape matches &n)
    "+"   -> "&op"  (operator shape matches &op)
    "cat" -> "&word" (word shape matches &word)

the output is a tuple: (rewritten_string, bindings) where bindings is a position-keyed dictionary of SigilBinding structs: 

struct SigilBinding
    position     :: Int      # 1-based position in rewritten string
    name         :: Symbol   # which sigil captured this
    value        :: String   # the original surface token
    class        :: Symbol   # sigil class
    surface      :: String   # pre-canonicalization form
    raw_position :: Int      # position in original (un-rewritten) input
end

the bindings travel as a side channel. the rewritten string drives pattern matching. the bindings drive computation. they are two views of the same input, separated by purpose.

2.2 the promote_input pipeline

function promote_input(raw_input, registry)
    # Phase 1: canonicalize
    canonical = raw_input
    for (word, symbol) in NUMBER_WORD_MAP
        canonical = replace(canonical, word => symbol)
    for (word, symbol) in OP_WORD_MAP
        canonical = replace(canonical, word => symbol)

    # Phase 2: tokenize + shape-promote
    tokens = split(canonical)
    rewritten = []
    bindings  = Dict{Int, SigilBinding}()

    for (i, token) in enumerate(tokens)
        if starts_with_ampersand(token)
            # already a sigil -- pass through (idempotency)
            push!(rewritten, token)
            continue

        # find best matching sigil by shape
        best_match = find_shape_match(token, registry)

        if best_match !== nothing && best_match.promote_at_tokenize
            # Phase 1.5c: promote_predicate gate
            if best_match.promote_predicate(token)
                push!(rewritten, "&" + string(best_match.name))
                bindings[i] = SigilBinding(
                    position = i,
                    name     = best_match.name,
                    value    = token,
                    class    = best_match.class,
                    surface  = raw_token_at(i),
                    raw_position = original_position(i)
                )
            else
                push!(rewritten, token)  # predicate said no
            end
        else
            push!(rewritten, token)  # no sigil match
        end

    return (join(rewritten, " "), bindings)

2.3 promote_predicate: the conditional gate

not every token that could match a sigil should be promoted. the promote_predicate field on each SigilEntry is a function that takes the surface token and returns true or false. this is stage 1.5c -- a conditional gate between shape-matching and actual rewriting.

example: &n has a promote_predicate that checks whether the token is actually numeric (not just looks numeric in certain contexts). &op checks whether the token is in the operator lexicon. this prevents over-eager promotion of ambiguous tokens. 

# &n promote_predicate
(token) -> occursin(r"^[0-9]+\.?[0-9]*$", token)

# &op promote_predicate
(token) -> token in ["+", "-", "*", "/", "^", "%"]

2.4 idempotency

if input is already promoted (contains &sigil tokens), the promoter passes them through unchanged. you can run promote_input on its own output and get the same result. this is critical for composability -- downstream systems can re-promote without corruption. 

# already-promoted input passes through
promote_input("&n &op &n") == ("&n &op &n", {})

3. PROCEDURE SIGILS

3.1 math-acronym macros

procedure sigils are ordered chains of sigil references and literals. think PEMDAS as a macro: you define a procedure sigil called &arith that expands into a sequence of evaluation steps. each step is either a literal operation or a reference to another sigil. 

# registration
register_procedure_sigil!(registry,
    name      = :arith,
    expansion = ["&n", "&op", "&n"],   # left-to-right bind order
    params    = Dict(:max_depth => 8)
)

the expansion list is not a flat string. it is an ordered vector of references. each reference is resolved at expansion time, meaning procedure sigils can reference other procedure sigils, producing recursive expansion trees bounded by MAX_PROCEDURE_DEPTH = 8.

3.2 recursive expansion

expand_procedure_sigil walks the expansion vector depth-first. when it encounters a sigil reference (a string starting with &), it looks up that sigil in the registry. if the referenced sigil is also a procedure, it recurses. if it is a lambda or macro, it returns the binding. if it is a literal string, it returns the literal. 

function expand_procedure_sigil(name, registry, bindings, depth=0)
    if depth > MAX_PROCEDURE_DEPTH
        error("procedure expansion exceeded max depth")

    entry = registry[name]
    result = []

    for target in entry.expansion
        if startswith(target, "&")
            ref_name = Symbol(target[2:end])
            ref_entry = registry[ref_name]

            if ref_entry.class == :procedure
                # recursive expansion
                sub = expand_procedure_sigil(ref_name, registry,
                                             bindings, depth + 1)
                append!(result, sub)
            elseif ref_entry.class == :lambda
                # pull from bindings
                pos = ref_entry.params[:position]
                push!(result, bindings[pos].value)
            elseif ref_entry.class == :macro
                # pull from bindings (first match)
                for (p, b) in bindings
                    if b.name == ref_name
                        push!(result, b.value)
                        break
            else
                push!(result, target)  # literal passthrough
        else
            push!(result, target)  # literal string

    return result

3.3 example: multi-step arithmetic procedure

define a procedure for evaluating "(a + b) * c": 

# step 1: register sub-procedure for the paren group
register_procedure_sigil!(registry,
    name      = :paren_sum,
    expansion = ["&n", "+", "&n"],   # binds positions 1 and 3
    params    = Dict(:position => 1)
)

# step 2: register the full expression procedure
register_procedure_sigil!(registry,
    name      = :expr_mul,
    expansion = ["&paren_sum", "*", "&n"],  # recursive ref + literal + lambda
    params    = Dict(:position => 2)
)

# expansion trace for input "3 + 5 times 2":
# promote_input -> "&n + &n * &n" with bindings {1=>3, 3=>5, 5=>2}
# expand_procedure_sigil(:expr_mul)
#   -> expand "&paren_sum" -> expand_procedure_sigil(:paren_sum)
#       -> ["3", "+", "5"]
#   -> "*"
#   -> bindings[5] = "2"
#   result: ["3", "+", "5", "*", "2"]
#   evaluate: (3 + 5) * 2 = 16

the depth bound prevents infinite loops from mutually-referencing procedures. at depth 8, expansion halts and returns whatever has been accumulated so far (plus an error flag). in practice, most useful procedures are depth 1-3.

4. RELATION SIGILS

4.1 dynamic relational triples

relation sigils expand to alternative verb strings for dynamic relational triples. a relational triple is (subject, verb, object). the relation sigil provides the verb slot with a set of alternatives drawn from a semantic category: 

(subject, &temporal, object) ->
  (subject, "before",    object)
  (subject, "after",     object)
  (subject, "during",    object)
  (subject, "preceding", object)
  (subject, "following", object)
  (subject, "while",     object)
  ...

the purpose is not to enumerate all possibilities but to give the matching engine a semantic category to work with. a node that matches on &temporal can fire on any temporal relation verb, without needing separate patterns for each one.

4.2 registration and expansion

function register_relation_sigil!(registry, name, expansion, provenance=:user)
    entry = SigilEntry(
        name       = name,
        class      = :relation,
        applies_at = :relation,
        sigil_type = :relation,
        lexicon    = expansion,     # the verb strings ARE the lexicon
        params     = Dict(),
        expansion  = expansion,
        provenance = provenance,
        promote_at_tokenize = false,
        promote_predicate   = (_ -> false)  # relations don't promote from input
    )
    register_sigil!(registry, entry)

function expand_relation_sigil(name, registry)
    entry = registry[name]
    assert entry.class == :relation
    return entry.expansion   # vector of verb strings

4.3 built-in relation categories

sigilcategorysample expansions
&temporaltime relationsbefore, after, during, preceding, following, while
&causalcause/effectcauses, results in, leads to, produces, enables, prevents
&spatialspace relationsabove, below, beside, inside, outside, between, through
&possessiveownershiphas, owns, contains, holds, carries, possesses
&similarityresemblanceresembles, mirrors, approximates, echoes, parallels

the matching engine uses expand_relation_if_sigil to check whether a verb token falls inside a relation sigil's expansion: 

function expand_relation_if_sigil(token, registry)
    for (name, entry) in registry
        if entry.class == :relation && token in entry.expansion
            return (true, name, entry.expansion)
    return (false, nothing, nothing)

this is how the system knows that "before" and "after" are both temporal relations, without hard-coding that knowledge anywhere outside the registry entry for &temporal.

5. ARITHMETIC ENGINE: A CONCRETE AUTOMATON

5.1 from bindings to numbers

the ArithmeticEngine is the first concrete automaton built on sigil bindings. it reads the &n and &op bindings produced by the promoter and evaluates the expression. the architecture: 

  user input: "what is two plus three times five"
      |
      v
  SigilPromoter.promote_input()
      |
      +-- rewritten: "what is &n &op &n &op &n"
      +-- bindings:  {2 => SigilBinding(:n, "2", "two"),
                     4 => SigilBinding(:op, "+", "plus"),
                     6 => SigilBinding(:n, "3", "three"),
                     8 => SigilBinding(:op, "*", "times"),
                     10=> SigilBinding(:n, "5", "five")}
      |
      v
  ArithmeticEngine.compute_arithmetic(bindings)
      |
      +-- steps: [Step(2, "+", 3), Step(5, "*", 5)]  // left-to-right
      +-- result: 25  (2+3=5, 5*5=25)
      |
      v
  format_arithmetic_reply(result, steps)
      -> "two plus three is five, times five is twenty-five"

5.2 ComputationStep struct

struct ComputationStep
    left     :: Float64
    operator :: String
    right    :: Float64
end

struct ArithmeticResult
    value      :: Float64
    steps      :: Vector{ComputationStep}
    expression :: String
    success    :: Bool
    error_msg  :: Union{String, Nothing}
end

5.3 multi-step evaluation

the engine evaluates left-to-right with no operator precedence. this is intentional -- the sigil binding order from the promoter is the evaluation order. if the user wants precedence, they use procedure sigils to define the grouping. 

function compute_arithmetic(bindings)
    nums = []   # collect &n bindings in order
    ops  = []   # collect &op bindings in order

    for (pos, binding) in sort(bindings)
        if binding.name == :n
            push!(nums, parse_float(binding.value))
        elseif binding.name == :op
            push!(ops, binding.operator)

    if length(ops) == 0 && length(nums) == 1
        return ArithmeticResult(nums[1], [], string(nums[1]), true, nothing)

    if length(ops) != length(nums) - 1
        return ArithmeticResult(0, [], "", false, "mismatched operators and operands")

    # left-to-right evaluation
    accumulator = nums[1]
    steps = []
    for i in 1:length(ops)
        step = ComputationStep(accumulator, ops[i], nums[i+1])
        accumulator = apply_op(accumulator, ops[i], nums[i+1])
        push!(steps, step)

    return ArithmeticResult(accumulator, steps, format_expr(nums, ops),
                           true, nothing)

5.4 natural language output

format_arithmetic_reply converts the computation back into spoken language, using the surface field from bindings to recover the user's original words: 

function format_arithmetic_reply(result, steps, bindings)
    parts = []
    for step in steps
        left_word  = find_surface_for_value(step.left, bindings)
        op_word    = find_surface_for_op(step.operator, bindings)
        right_word = find_surface_for_value(step.right, bindings)
        result_word = number_to_words(step.result)

        push!(parts, left_word + " " + op_word + " " + right_word +
                     " is " + result_word)

    return join(parts, ", ")
    # e.g. "two plus three is five, five times five is twenty-five"

this round-trip -- natural input to sigil bindings to computation back to natural output -- is the full automaton lifecycle in miniature.

6. FAST MAPPING VIA SIGILS

6.1 the cognitive phenomenon

in cognitive science, fast mapping is the ability of children (and adults) to learn the meaning of a new word after a single exposure, by slotting it into an existing syntactic/semantic structure. the key insight is that the structure already exists -- the learner has a prior that constrains where the new item can go.

sigils are that prior. a &n slot in a pattern says "a number goes here". when a new number appears in that slot, it is immediately bound with full type information, full position metadata, and full provenance. no statistical discovery. no gradient descent. one exposure, full integration.

6.2 sigils as structural priors

the analogy is exact: 

  cognitive fast mapping:     child hears "the dax is red"
                              knows "the ___ is red" structure
                              slots "dax" into the noun position
                              immediately knows dax is a noun-class thing

  sigil fast mapping:         system sees "what is xtal plus 3"
                              knows "&n &op &n" structure
                              "xtal" matches &n by shape
                              immediately bound as SigilBinding(:n, "xtal", ...)
                              full type, position, surface captured

the sigil registry is the "existing syntactic/semantic structure" that makes fast mapping possible. every new token is either:

  1. matched by an existing sigil (fast mapped, zero-shot learned)
  2. not matched (deferred to slower statistical or fallback processing)

6.3 extending the prior

when a user registers a new sigil, they are extending the structural prior. this is analogous to a child learning a new grammatical construction and then being able to fast-map tokens into it: 

# before: no sigil for hex numbers
# "what is 0xFF plus 0x0A" -> "0xFF" doesn't match &n, falls through

# user registers hex sigil:
register_sigil!(registry, SigilEntry(
    name       = :hex,
    class      = :lambda,
    applies_at = :bind,
    sigil_type = :numeric,
    lexicon    = ["0x0","0x1",...,"0xF","0xFF",...],
    params     = Dict(:position => 1, :base => 16),
    expansion  = [],
    provenance = :user,
    promote_at_tokenize = true,
    promote_predicate   = (t -> occursin(r"^0x[0-9A-Fa-f]+$", t))
))

# after: "what is 0xFF plus 0x0A" -> 0xFF fast-mapped as &hex
# arithmetic engine can now handle hex via &hex bindings

one registration. instant fast mapping for all inputs matching the new shape.

7. TIME SIGILS AND TEMPORAL AUTOMATA

7.1 time nodes are tense, not chronology

time nodes in the grug architecture are not about when things happened. they are about orientation toward action. three orientations: 

  PAST    = reflect   (look backward, summarize, recall)
  PRESENT = assess    (look around, evaluate, describe current state)
  FUTURE  = project   (look forward, predict, plan)

a user asking "what now?" hits a PRESENT-oriented time node. "what happened?" hits PAST. "what will happen?" hits FUTURE. the node type determines not the temporal fact but the reasoning mode the system enters.

7.2 reserved vote flags

time nodes carry reserved vote flags that command the AIML reasoning layer. when a time node fires in a cascade, its vote flags travel with the signal and instruct downstream processing how to orient: 

const TIME_ORIENTATIONS = Dict(
    :past    => Dict(:reflect => true,  :assess => false, :project => false),
    :present => Dict(:reflect => false, :assess => true,  :project => false),
    :future  => Dict(:reflect => false, :assess => false, :project => true)
)

# when a time node fires, its vote includes:
vote.flags = TIME_ORIENTATIONS[node.orientation]
# downstream AIML reads vote.flags[:reflect] etc.

7.3 time sigil macros

the time sigil system lets end users define temporal automata using the same macro expansion machinery. a time sigil is a procedure sigil whose expansion wires vote flags and seam tokens automatically: 

# register a temporal macro for "what happens next" -> FUTURE orientation
register_procedure_sigil!(registry,
    name      = :t_next,
    expansion = ["&future_orient", "&assess_current", "&project_next"],
    params    = Dict(
        :orientation => :future,
        :vote_flags  => Dict(:reflect => false, :assess => true, :project => true),
        :seam_tokens => ["<NOW>", "<NEXT>"]
    )
)

# expansion produces an automaton sequence:
#   1. &future_orient  -> sets node orientation to :future, wires vote flags
#   2. &assess_current -> triggers AIML assessment of current state
#   3. &project_next   -> triggers AIML projection of next state
#   seam tokens <NOW> and <NEXT> bridge between steps 2 and 3

7.4 coherence-ordered transitions

temporal automata transition between states in coherence order, not chronological order. coherence means: the next state is the one that maximally resolves the current state's open questions. this is computed by checking which remaining expansion step has the highest overlap with already-resolved bindings: 

function next_temporal_state(current, remaining_steps, resolved_bindings)
    best_step = nothing
    best_coherence = -1

    for step in remaining_steps
        # compute overlap between step's required bindings and resolved
        coherence = count_intersect(step.required_bindings, resolved_bindings)
        if coherence > best_coherence
            best_coherence = coherence
            best_step = step

    return best_step

this means temporal automata are not fixed sequences. they are opportunistic -- they follow the path of maximum coherence given what has already been resolved. if the user provides information out of order, the automaton adapts.

7.5 user-defined temporal automata

the end user can define arbitrary temporal automata by registering procedure sigils with :orientation parameters: 

# "morning routine" automaton
register_procedure_sigil!(registry,
    name      = :morning_routine,
    expansion = ["&past_orient", "&reflect_sleep",
                 "&present_orient", "&assess_energy",
                 "&future_orient", "&plan_day"],
    params    = Dict(
        :orientation => :sequential,  # past->present->future in order
        :vote_flags  => Dict(:reflect => true, :assess => true, :project => true),
        :seam_tokens => ["<WOKE>", "<ENERGY>", "<PLAN>"]
    )
)

# "troubleshooting" automaton (present-centric with future probes)
register_procedure_sigil!(registry,
    name      = :troubleshoot,
    expansion = ["&present_orient", "&assess_state",
                 "&present_orient", "&assess_logs",
                 "&future_orient", "&project_fix"],
    params    = Dict(
        :orientation => :present_heavy,
        :vote_flags  => Dict(:reflect => false, :assess => true, :project => true)
    )
)

each macro expansion auto-wires the vote flags and seam tokens. the user just declares the sequence. the system handles the wiring.

8. SIGIL-MACRO LLM: REPLACING TRANSFORMERS

8.1 the core thesis

a transformer does three things: it attends over context, it interpolates from training data, and it predicts the next token. a sigil-macro system does three different things: it matches input to structural priors, it expands those priors into automaton instructions, and it executes those instructions to produce output. the mapping: 

  TRANSFORMER              SIGIL-MACRO
  -----------              -----------
  training      =          definition (register_sigil!)
  inference     =          execution  (expand + compute)
  context       =          state      (bindings + registry)
  attention     =          matching   (shape-promote + resolve)
  parameters    =          entries    (SigilEntry structs)
  gradient      =          (none)     (no optimization needed)

8.2 architecture

                        +-----------------+
                        |   USER INPUT    |
                        +--------+--------+
                                 |
                                 v
                        +--------+--------+
                        |  SIGIL PROMOTER  |  canonicalize + shape-promote
                        +--------+--------+
                                 |
                         rewritten + bindings
                                 |
                                 v
                        +--------+--------+
                        |  SIGIL MATCHER   |  resolve_sigils_in_pattern
                        +--------+--------+        |
                                 |           matched sigil entries
                                 v
                        +--------+--------+
                        |  MACRO EXPANDER  |  expand_procedure/relation_sigil
                        +--------+--------+
                                 |
                          automaton instructions
                                 |
                                 v
                        +--------+--------+
                        |  AUTOMATON EXEC  |  compute / render / respond
                        +--------+--------+
                                 |
                                 v
                           +-----+------+
                           |   OUTPUT   |
                           +------------+

8.3 training is definition

in a transformer, "learning" means adjusting billions of floating point weights via backpropagation. in a sigil-macro LLM, "learning" means calling register_sigil!: 

# "teach" the system about color adjectives

    class      = :macro,
    applies_at = :bind,
    sigil_type = :alphanumeric,

                  "cyan","magenta","black","white","gray","brown"],
    params     = Dict(),
    expansion  = [],
    provenance = :user,
    promote_at_tokenize = true,

# every input containing a color adjective is now fast-mapped.

8.4 inference is execution

when input arrives, the system does not interpolate between training examples. it executes the automaton defined by the matched sigil's expansion. the output is deterministic given the input and the registry state: 

# input: "the red cat sat on the blue mat"
# promotion: "the &color &noun sat on the &color &noun"
# bindings: {2=>(:color,"red"), 3=>(:noun,"cat"),
#            7=>(:color,"blue"), 8=>(:noun,"mat")}
#
# match: two color_noun procedure instances
# expand: [&color, &noun] for each instance
# execute: bind and produce structured representation
#
# result: [ColorNoun("red","cat"), ColorNoun("blue","mat")]
#
# no interpolation. no hallucination. pure execution.

8.5 context is state

a transformer maintains context through attention over a sliding window of tokens. a sigil-macro system maintains context through the registry state + accumulated bindings. the registry is the long-term context (which sigils exist). the bindings are the short-term context (what the current input captured). together they are the full context, no attention matrix required. 

# long-term context: the registry
# (persists across conversations, grows as user adds sigils)

# short-term context: the bindings
# (created fresh for each input, consumed by the automaton)

# conversation state = registry state + current bindings
# no KV cache. no context window. no token limit.

8.6 ephemeral automata

automata spawned by macro expansion are ephemeral. they exist for one input cycle. they compute their result. they vanish. no persistent weights. no memory of past executions (unless explicitly stored in the registry). this makes the system naturally stateless at the automaton layer, which is a feature not a bug:

  • no stale state accumulation
  • no context window overflow
  • no cross-contamination between inputs
  • deterministic replay given same registry + input
# each input cycle:
1. promote_input(raw) -> (rewritten, bindings)
2. match(rewritten, registry) -> matched_entries
3. expand(matched_entries, registry, bindings) -> instructions
4. execute(instructions) -> output
5. discard bindings, discard instructions, discard automaton
6. keep registry (it persists)
7. keep output (it is the answer)

9. TEXT RENDERING VIA SIGIL AUTOMATA

9.1 the observation

operating systems render text by rasterizing each glyph from a font atlas, compositing the results into a framebuffer. for a 1000-word document with an average of 5 characters per word, that is 5000 rasterization operations. most of those glyphs are repeated -- "e" alone accounts for roughly 13% of English text. the redundancy is massive.

9.2 sigil-based rendering

the idea: define sigils for paragraph, word, and character-class levels. each sigil expands into a render automaton that maintains state between glyphs. repeated glyphs hit the same automaton state, which returns a cached bitmap instead of re-rasterizing:

9.3 the render pipeline

function sigil_render(text, registry, framebuffer, cursor)
    # promote: classify each character into glyph sigils
    (rewritten, bindings) = promote_render_input(text, registry)

    for (pos, binding) in sort(bindings)
        entry = registry[binding.name]

        # check glyph cache in entry.params[:cache]
        ch = binding.value[1]   # single character
        if haskey(entry.params[:cache], ch)
            # cache hit: blit cached bitmap, advance cursor
            bitmap = entry.params[:cache][ch]
            blit(framebuffer, cursor, bitmap)
            advance(cursor, bitmap.width)
        else
            # cache miss: rasterize, store, blit, advance
            bitmap = rasterize(ch, current_font, current_size)
            entry.params[:cache][ch] = bitmap
            blit(framebuffer, cursor, bitmap)
            advance(cursor, bitmap.width)

    return framebuffer

higher-level sigils handle word wrapping, line breaks, and paragraph spacing. these are procedure sigils that expand into sequences of lower-level glyph sigils plus layout instructions:

9.5 constant-cost state transitions

the key performance property: for a glyph that has been seen before, the render automaton does constant work. it checks the cache (O(1) dict lookup), blits the bitmap, and advances the cursor. no font lookup, no hinting, no rasterization. the cost of rendering "e" the 500th time is the same as the cost of rendering it the 2nd time (the 1st time pays the rasterization). 

# cost model:
#   first occurrence of glyph g:  O(rasterize) + O(1)  =  O(rasterize)
#   subsequent occurrences of g:  O(1) cache hit + O(1) blit = O(1)
#
# for a document with N total characters, U unique characters:
#   total cost = U * O(rasterize) + (N - U) * O(1)
#             = O(U * rasterize + N)
#
# compare to traditional: N * O(rasterize) = O(N * rasterize)
#
# speedup factor: N * rasterize / (U * rasterize + N)
#               ≈ N / U for large N (since rasterize >> 1)
#
# for English text: N ≈ 5000, U ≈ 70 (52 letters + 10 digits + ~8 punct)
# speedup ≈ 5000 / 70 ≈ 71x

this is not a minor optimization. it is a structural change in how text rendering works. the sigil cache makes rendering an O(N + U) operation instead of O(N * R) where R is the per-glyph rasterization cost.

10. TOPOLOGICAL FOLDING AND SIGIL ROUTING

10.1 primes as folds

the sieve of eratosthenes can be understood as a topological operation. each prime p "folds" the number line by wrapping it into a loop (Z/pZ -- the integers modulo p). the first fold (p=2) is unique: it bipartitions the entire number line into even and odd, setting the floor for the minimum gap between surviving numbers. every subsequent fold operates on a line already bent by all previous folds.

the gap distribution between consecutive primes encodes the history of all folds applied so far. this is why twin primes (gap=2) cluster where the fold history has left room for a minimal gap -- the residual of the p=2 bipartition that hasn't been consumed by higher folds.

10.2 folding as sigil routing

the same topological intuition applies to sigil routing. each sigil class defines a "fold" on the input token stream: 

  &n fold:    splits tokens into numeric / non-numeric
  &op fold:   splits tokens into operator / non-operator
  &word fold: splits tokens into word / non-word
  &temporal fold: splits relations into temporal / non-temporal

the order of folds matters, just as the order of prime folds matters. the &n fold (numeric bipartition) is the first fold applied by the promoter, analogous to p=2 being the first sieve fold. subsequent folds operate on a token stream already partitioned by previous sigil matches.

10.3 routing affinity

the concept of routing affinity captures how strongly a token is drawn to a particular sigil fold. a token like "42" has high affinity for &n and low affinity for everything else. a token like "before" has high affinity for &temporal but also moderate affinity for &word. the promote_predicate resolves these affinities by deciding which fold the token enters: 

function compute_affinities(token, registry)
    affinities = Dict{Symbol, Float64}()
    for (name, entry) in registry
        if entry.promote_at_tokenize && entry.promote_predicate(token)
            # base affinity from lexicon membership
            if token in entry.lexicon
                affinities[name] = 1.0
            else
                # shape-based affinity (partial match)
                affinities[name] = shape_similarity(token, entry.sigil_type)
    return affinities

# "before" affinities:
#   &temporal : 1.0  (lexicon member)
#   &word     : 0.8  (shape match: alphanumeric word)
#   &n        : 0.0  (not numeric)
#
# promote_predicate for &temporal says true
# promote_predicate for &word says true
# winner: &temporal (highest affinity)
# "before" enters the &temporal fold

10.4 oscillation vs routing

a transformer resolves ambiguity through oscillation -- multiple attention heads "debate" the classification of a token over many layers. a sigil system resolves ambiguity through routing -- the token enters exactly one fold, determined by affinity ranking. there is no debate. the first fold wins. subsequent folds see only the residue.

this is why biological morphogenesis works the same way. cell fate determination is not a committee vote. it is a routing decision. the first morphogen gradient that crosses threshold wins the cell. all subsequent gradients see a cell already committed. topological folding over a developmental surface, routing affinity over oscillation. same math, different substrate.

11. AUTOMATON COMPOSITION PATTERNS

11.1 sequential composition

the simplest pattern: procedure A's output feeds procedure B's input. this is the default behavior of procedure sigil expansion: 

# define a two-stage pipeline
register_procedure_sigil!(registry,
    name      = :parse_and_eval,
    expansion = ["&parse_expr", "&eval_expr"],
    params    = Dict(:position => 1)
)

# expansion:
#   1. &parse_expr binds the input tokens into a structured form
#   2. &eval_expr takes the structured form and computes the result
#   seam between them: the bindings from step 1 are available to step 2

11.2 parallel composition

multiple sigils match the same input region. their expansions run in parallel and the results are merged by a voting mechanism: 

# input: "the big red cat"
# matches: &adjective at "big", &adjective at "red", &noun at "cat"
# expansions run in parallel:
#   &adjective("big") -> [AdjEntry("big", :size)]
#   &adjective("red") -> [AdjEntry("red", :color)]
#   &noun("cat")      -> [NounEntry("cat", :animal)]
# merge: [AdjEntry, AdjEntry, NounEntry] -> NounPhrase("big red cat")

11.3 conditional composition

the promote_predicate acts as a conditional gate. different inputs take different paths through the same sigil, depending on whether the predicate fires: 

# &n with a conditional that rejects hex values:
#   input "42"  -> predicate true  -> promoted to &n -> arithmetic path
#   input "0xFF" -> predicate false -> not promoted  -> fallback path
#
# if &hex sigil is registered later:
#   input "0xFF" -> &hex predicate true -> promoted to &hex -> hex arithmetic path

11.4 recursive composition

procedure sigils that reference other procedure sigils. bounded by MAX_PROCEDURE_DEPTH. the recursion is not unbounded -- it is a controlled unfolding of nested definitions: 

# recursive arithmetic with precedence
register_procedure_sigil!(registry,
    name      = :expr,
    expansion = ["&term", "&add_op", "&expr"],  # right-recursive
    params    = Dict(:max_depth => 8)
)

register_procedure_sigil!(registry,
    name      = :term,
    expansion = ["&factor", "&mul_op", "&term"],  # right-recursive
    params    = Dict(:max_depth => 8)
)

register_procedure_sigil!(registry,
    name      = :factor,
    expansion = ["&n"],  # base case
    params    = Dict()
)

# expansion of "2 + 3 * 4":
#   &expr -> [&term, &add_op, &expr]
#     &term -> [&factor, &mul_op, &term]
#       &factor -> [&n] -> [2]
#       &mul_op -> [*]
#       &term -> [&factor] -> [&n] -> [4]
#     result: 3 * 4 = 12
#     &add_op -> [+]
#     &expr -> [&term] -> [&factor] -> [&n] -> [12]
#   result: 2 + 12 = 14

11.5 ephemeral vs persistent automata

by default, automata are ephemeral -- they execute once and vanish. but some applications need persistent automata that maintain state across inputs. the registry itself serves as the persistence layer: 

# persistent counter automaton
register_procedure_sigil!(registry,
    name      = :count_entity,
    expansion = ["&entity", "increment_counter"],
    params    = Dict(
        :counter => Ref(0),  # mutable reference persists in registry
        :entity_name => nothing
    )
)

# each execution:
#   1. &entity binds the matched entity
#   2. increment_counter reads and increments params[:counter][]
#   counter value persists across inputs because it lives in the registry

12.2 the design principles

1. symbols are active, not passive. a sigil is not a label. it is a machine. its expansion is its program. its parameters are its memory. its predicate is its guard condition.

2. promotion is rewriting. input is not processed in place. it is rewritten into a canonical form that the downstream system can match on. the original form is preserved in the bindings side channel.

3. matching is routing. a token enters exactly one sigil fold. ambiguity is resolved by affinity ranking, not by committee. the first fold that accepts the token owns it.

4. expansion is compilation. macro expansion converts sigil references into executable instructions. recursive expansion handles nested definitions. the depth bound prevents infinite loops.

5. execution is deterministic. given the same registry state and the same input, the system produces the same output. no randomness. no sampling temperature. no top-k. the answer is the answer.

6. state is the registry. everything that persists lives in the registry. everything that is ephemeral lives in bindings and instructions. the boundary is clear. the registry is the long-term memory. the bindings are the working memory.

7. growth is additive. new capabilities are added by registering new sigils. existing capabilities are never broken by additions. the registry is a monoid: the empty registry is the identity, and merge is the operation.

12.3 what this enables

the sigil-macro-automata architecture is not just a different way to build chatbots. it is a different way to think about symbolic computation. every domain where structured input meets structured processing is a candidate: 

  arithmetic       - done (ArithmeticEngine)
  natural language - done (promoter + matching + formatting)
  temporal reason  - ready (time sigils + vote flags)
  text rendering   - ready (glyph sigils + cache)
  code parsing     - obvious (token class sigils + procedure expansion)
  query planning   - obvious (relation sigils + coherent ordering)
  game AI          - possible (state sigils + ephemeral strategy automata)
  OS shell         - possible (command sigils + pipeline procedures)

the common pattern: define the symbols, define their expansions, register them, and the system can process inputs in that domain. no training. no data. no GPU. just definitions.

A.1 complete promote_input

function promote_input(raw_input, registry)
    # === LAYER 1: CANONICALIZE ===
    canonical = lowercase(raw_input)
    canonical = strip(canonical)

    # number words -> digits
    for (word, digit) in NUMBER_WORD_MAP   # "zero"->"0", ... "nine"->"9"
        canonical = replace(canonical, r"\b" + word + r"\b" => digit)
    # multi-word numbers: "twenty one" -> "21"
    # (handled by composite number parser after basic word map)

    # operator words -> symbols
    for (word, symbol) in OP_WORD_MAP     # "plus"->"+", "minus"->"-", ...
        canonical = replace(canonical, r"\b" + word + r"\b" => symbol)

    # === LAYER 2: SHAPE-PROMOTE ===
    tokens = split(canonical)
    rewritten_tokens = []
    bindings = Dict{Int, SigilBinding}()
    sigil_counts = Dict{Symbol, Int}()  # track position per sigil

    for (i, token) in enumerate(tokens)
        if startswith(token, "&")
            # already promoted -- idempotent passthrough
            push!(rewritten_tokens, token)
            name = Symbol(token[2:end])
            sigil_counts[name] = get(sigil_counts, name, 0) + 1
            continue

        # find best sigil match by shape
        best = find_best_shape_match(token, registry)

        if best !== nothing && best.promote_at_tokenize
            # conditional gate
            if best.promote_predicate(token)
                name = best.name
                sigil_counts[name] = get(sigil_counts, name, 0) + 1

                push!(rewritten_tokens, "&" + string(name))

                bindings[i] = SigilBinding(
                    position     = sigil_counts[name],
                    name         = name,
                    value        = token,
                    class        = best.class,
                    surface      = find_surface(i, raw_input),
                    raw_position = find_raw_position(i, raw_input)
                )
                continue

        # no match or predicate rejected -- pass through as-is
        push!(rewritten_tokens, token)

    return (join(rewritten_tokens, " "), bindings)

A.2 complete expand_procedure_sigil

const MAX_PROCEDURE_DEPTH = 8

function expand_procedure_sigil(name, registry, bindings, depth=0)
    if depth > MAX_PROCEDURE_DEPTH
        return ExpansionResult([], false, "max depth exceeded")

    if !haskey(registry, name)
        return ExpansionResult([], false, "sigil not found: " + name)

    entry = registry[name]

    if entry.class != :procedure
        return ExpansionResult([], false, "not a procedure sigil: " + name)

    result = []
    all_ok = true

    for target in entry.expansion
        if startswith(target, "&")
            ref_name = Symbol(target[2:end])

            if !haskey(registry, ref_name)
                all_ok = false
                continue

            ref_entry = registry[ref_name]

            switch ref_entry.class
                case :procedure
                    sub = expand_procedure_sigil(ref_name, registry,
                                                 bindings, depth + 1)
                    if sub.success
                        append!(result, sub.elements)
                    else
                        all_ok = false

                case :lambda
                    # find binding for this lambda at current position
                    pos = ref_entry.params[:position]
                    for (i, b) in bindings
                        if b.name == ref_name && b.position == pos
                            push!(result, b.value)
                            break

                case :macro
                    # find first binding for this macro
                    for (i, b) in bindings
                        if b.name == ref_name
                            push!(result, b.value)
                            break

                case :relation
                    verbs = expand_relation_sigil(ref_name, registry)
                    push!(result, verbs)  # attach full alternation set

                default
                    push!(result, target)

        else
            # literal string
            push!(result, target)

    return ExpansionResult(result, all_ok, nothing)

A.3 complete compute_arithmetic

function compute_arithmetic(bindings)
    # collect numeric and operator bindings in position order
    sorted = sort(collect(bindings), by = (pair -> pair[1]))

    nums = []
    ops  = []

    for (pos, binding) in sorted
        if binding.name == :n
            push!(nums, parse(Float64, binding.value))
        elseif binding.name == :op
            push!(ops, binding.value)

    # validation
    if length(nums) == 0
        return ArithmeticResult(0, [], "", false, "no numbers found")

    if length(nums) == 1 && length(ops) == 0
        return ArithmeticResult(nums[1], [], string(nums[1]), true, nothing)

    if length(ops) != length(nums) - 1
        return ArithmeticResult(0, [], "", false,
                 "operator/operand count mismatch: " +
                 string(length(ops)) + " ops, " +
                 string(length(nums)) + " nums")

    # left-to-right evaluation (no operator precedence)
    accumulator = nums[1]
    steps = []

    for i in 1:length(ops)
        right = nums[i + 1]
        op = ops[i]

        result = apply_op(accumulator, op, right)
        if result === nothing
            return ArithmeticResult(0, [], "", false,
                     "unknown operator: " + op)

        push!(steps, ComputationStep(accumulator, op, right))
        accumulator = result

    return ArithmeticResult(accumulator, steps,
                           format_expression(nums, ops),
                           true, nothing)

function apply_op(left, op, right)
    switch op
        case "+"  return left + right
        case "-"  return left - right
        case "*"  return left * right
        case "/"  return right != 0 ? left / right : nothing
        case "^"  return left ^ right
        case "%"  return right != 0 ? left % right : nothing
        default   return nothing

A.4 time sigil auto-wiring

function auto_wire_temporal_macro!(registry, name, orientation, steps)
    # build expansion vector from step names
    expansion = []
    for step in steps
        push!(expansion, "&" + string(step))
    end

    # build vote flags from orientation
    vote_flags = Dict(
        :reflect => (orientation in [:past, :sequential]),
        :assess  => (orientation in [:present, :sequential, :present_heavy]),
        :project => (orientation in [:future, :sequential, :present_heavy])
    )

    # build seam tokens between steps
    seam_tokens = []
    for i in 1:length(steps)-1
        push!(seam_tokens, "<" + uppercase(steps[i]) + "_" +
                           uppercase(steps[i+1]) + ">")

    register_procedure_sigil!(registry,
        name      = name,
        expansion = expansion,
        params    = Dict(
            :orientation  => orientation,
            :vote_flags   => vote_flags,
            :seam_tokens  => seam_tokens,
            :auto_wired   => true
        )
    )
end

struct GlyphCache
    bitmaps   :: Dict{Char, Bitmap}     # char -> rasterized bitmap
    hits      :: Dict{Char, Int}        # char -> hit count (diagnostics)
    misses    :: Dict{Char, Int}        # char -> miss count (diagnostics)
    font_id   :: String                 # font + size identifier
    max_size  :: Int                     # max cache entries (eviction policy)
end

function render_with_cache(text, cache, framebuffer, cursor)
    for ch in text
        if haskey(cache.bitmaps, ch)
            # CACHE HIT: constant cost
            cache.hits[ch] = get(cache.hits, ch, 0) + 1
            bitmap = cache.bitmaps[ch]
        else
            # CACHE MISS: pay rasterization cost
            cache.misses[ch] = get(cache.misses, ch, 0) + 1
            bitmap = rasterize_glyph(ch, cache.font_id)

            # eviction: LRU if cache is full
            if length(cache.bitmaps) >= cache.max_size
                evict_lru!(cache)

            cache.bitmaps[ch] = bitmap

        blit(framebuffer, cursor.x, cursor.y, bitmap)
        cursor.x += bitmap.advance_width

    return framebuffer

function render_stats(cache)
    total_hits   = sum(values(cache.hits))
    total_misses = sum(values(cache.misses))
    total_ops    = total_hits + total_misses
    hit_rate     = total_hits / total_ops
    traditional_ops = total_ops   # every char = rasterize
    sigil_ops    = total_misses  # only misses pay rasterize cost
    speedup      = traditional_ops / max(sigil_ops, 1)

    return Dict(
        :hit_rate    => hit_rate,
        :speedup     => speedup,
        :unique_glyphs => length(cache.bitmaps),
        :total_chars => total_ops
    )
