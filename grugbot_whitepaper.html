1. Overview & Philosophy

GrugBot is a neuromorphic AI engine written in Julia that models cognition through competing node populations rather than traditional neural networks. Each node is an independent pattern-matcher that votes for actions based on input similarity. The system exhibits emergent behavior through stochastic competition, strength-based natural selection, and cross-domain cascade activation across specialized brain lobes.

Core Design Principles

  • No Silent Failures — Every error is surfaced loudly. All catch blocks log via @warn or rethrow. Zero bare catch blocks.
  • Stochastic Competition — Nodes compete via weighted coinflips, not deterministic ranking. The @coinflip macro drives all probabilistic decisions.
  • Biological Metaphors — Lobes partition knowledge domains. BrainStem dispatches signals. Phagy cycles clean dead nodes. The prefrontal cortex (AIML) orchestrates cross-domain reasoning.
  • GRUG Comment Style — All code uses inline cave-speak comments (# GRUG:) plus academic footer blocks for formal documentation.
  • Thread Safety — All shared state protected by ReentrantLock. Node map, queues, and thesauri are thread-safe.
  • Ambient Field / Instant Activation — The node population is never fully off. The signal is not transmitted — it is already present. Intensity rise is instant because the field is pre-aligned. See §17.5.
The architecture draws from Hopfield networks (associative memory caching), competitive learning (strength-biased selection), apoptosis (grave markers for failed nodes), and winner-take-all lateral inhibition (BrainStem dispatch). The AIML orchestrator functions as a prefrontal cortex — a centralized executive that integrates votes from all specialized cortical regions (lobes) into a unified response. The always-on ambient field and resolution-latency model (§17.5) mean that familiar inputs resolve near-instantly via Hopfield cache, while novel inputs trigger full stochastic competition — mirroring biological attention and rest-state consolidation.

Grug made binary. One file. You run it. That's it. No compile, no dependency hell, no "just install these 47 packages first". Download rock, throw rock, rock works.

Prebuilt Binary (Recommended)

The fastest way to run GrugBot is the prebuilt self-extracting binary, built with bindboss. It packs the entire Julia source tree into a single ELF executable with built-in dependency checking.

# 1. Download the binary from the grug-binary/ directory
wget https://raw.githubusercontent.com/marshalldavidson61-arch/grugbot420/main/grug-binary/grugbot420

# 2. Make it executable
chmod +x grugbot420

# 3. Run it
./grugbot420

What happens on first run

  1. The binary checks for julia on your PATH
  2. If Julia is missing, it prints the download URL and exits cleanly
  3. If Julia is found, it extracts the source to ~/.bindboss/grugbot420/ and launches
  4. Subsequent runs skip extraction (persisted) and go straight to the Brain > prompt

Requirements

  • Julia 1.9+ — https://julialang.org/downloads/
  • Linux x86_64 (Debian/Ubuntu recommended)
  • ~50 MB free disk space for extraction cache

Resetting the extraction cache

# If you need a fresh extraction (e.g. after updating the binary):
rm -rf ~/.bindboss/grugbot420/
./grugbot420
The binary is built with bindboss v0.2.0, which uses a v2 trailer format with SHA-256 integrity hashing and optional Ed25519 signing. The stub is a statically-linked Go binary (~3.4 MB) that handles extraction, dependency checking, and launch. The packed payload is the complete src/ directory, gzip-compressed inside the executable.

Strength-Biased Selection

When scanning, each alive node gets a coinflip weighted by its strength:

P(activate) = 0.1 + 0.8 × (strength / STRENGTH_CAP)

Strength 0.0 → 10% chance. Strength 10.0 → 90% chance. This creates Darwinian selection pressure.

Selective Pattern Scan v7.3

Pattern scanning uses a two-factor tier selection system. The base tier is determined by screen_input_complexity() which examines the user input signal length and relational triple count. But each node's scan tier is further adjusted by _effective_scan_mode() based on the node's own pattern complexity — simple patterns get cheaper scans regardless of input complexity.

Input complexity → base tier:
score = (signal_length × 0.15) + (triple_count × 1.5)
score < 1.5 → tier 1 (cheap_scan)
score < 4.5 → tier 2 (medium_scan)
score ≥ 4.5 → tier 3 (high_res_scan)

Per-node pattern complexity → tier cap:
node signal ≤ 3 tokens → capped at tier 1
node signal 4–8 tokens → capped at tier 2
node signal > 8 tokens → no cap

The tier can only go down, never up. A complex input won't force a 2-token node through high-res two-pass scanning — the variance penalty would be numerically meaningless on such a short signal. This prevents O(n²) work on patterns too small to benefit from it, while ensuring complex patterns get the full discriminative power of high-res scanning when the input warrants it.

4. Action Packets v2 (Pipe-Delimited) v7

Action packets define what a node does when it wins a vote. Version 2 uses pipe-delimited format with inline negatives per action, allowing action names to contain commas.

Format

# Single action, default weight (1.0)
"greet"

# Single action with weight
"reason^3.5"

# Single action with inline negatives and weight
"greet[dont frown, dont insult]^2"

# Multiple actions, pipe-delimited
"greet^3 | reason^2 | analyze^1"

# Full format: pipe-delimited with mixed negatives
"greet[dont frown, dont insult]^3 | welcome[dont be rude]^2 | smile^1"

# Action names CAN contain commas (pipes are the delimiter)
"plan, execute^2 | greet^1"

Parser Return Values

function parse_action_packet(packet::String)
    → (positives, all_negatives, action_items)

    positives    :: Vector{Tuple{String, Float64}}
    # (action_name, weight) pairs for weighted selection

    all_negatives :: Vector{String}
    # Deduped union of ALL negatives across all actions

    action_items :: Vector{Tuple{String, Float64, Vector{String}}}
    # (action_name, weight, per_action_negatives) — full detail
end

Weighted Selection

select_action(packet) normalizes weights to probabilities and rolls a Categorical distribution via @coinflip:

P(action_i) = weight_i / Σ(weights)

For "heavy^10 | light^1": P(heavy) ≈ 91%, P(light) ≈ 9%

Inline Negatives

Negatives tell Grug what NOT to do. Each action can have its own "dont" list inside square brackets. When greet says [dont frown, dont insult], those are constraints passed to the AIML generator so the response avoids those behaviors. All negatives from all actions are also collected into a flat list for backward compatibility with the vote system.
⚠ Migration from v3: Old comma-delimited format "reason^2,analyze^1" is no longer valid. Use pipe-delimited: "reason^2 | analyze^1"

7. BrainStem: Winner-Take-All Dispatch

The BrainStem implements lateral inhibition across lobes. When multiple lobes produce competing responses, the BrainStem selects a winner based on confidence and propagates inhibitory signals to losers.

Dispatch Flow

  1. Each lobe's top-scoring node submits a bid (confidence score)
  2. BrainStem selects highest confidence bid as winner
  3. Winner's lobe gets a propagation signal (reinforcement)
  4. Losing lobes get inhibitory feedback
This models the thalamo-cortical loop in biological brains where the thalamus (BrainStem) gates which cortical region (lobe) gets to dominate the current processing cycle. The inhibitory propagation prevents runaway excitation and ensures focused responses.

8. Prefrontal Cortex: AIML Orchestration v7

The AIML orchestrator is GrugBot's prefrontal cortex — a centralized executive that integrates votes from ALL lobes into a unified response. It is NOT per-lobe; it sees the entire cognitive state.

Orchestration Pipeline

  1. Vote Collection: All specimens from all lobes cast votes
  2. Confidence Sort: Votes sorted by confidence descending (highest-confidence domain node wins primary)
  3. Bucketing: Sure votes (within 0.05 of max) vs. Unsure votes (50/50 coinflip)
  4. Lobe Context: extract_lobe_aware_context(votes) maps votes → lobes, builds cross-domain awareness
  5. Rule Evaluation: Stochastic AIML rules fire based on their probability, with template variables expanded
  6. Response Generation: JIT text synthesis combining all context layers

8.5 AIML Node System: Per-Lobe Tribes v7.6

The AIML Node System provides per-lobe "tribes" of AIML nodes with isolated populations and independent population caps. Each lobe can host its own AIML tribe, separate from other lobes.

Per-Lobe Tribes

When a lobe is registered for AIML via register_lobe!(lobe_id, node_cap), the system:

  • Creates an entry in AIML_REGISTRY keyed by lobe_id
  • Sets the AIML population cap to node_cap ÷ 3 (one-third of the lobe's node cap)
  • Initializes an empty tribe (Dict mapping node_id → AIMLNode)

Population Cap with Grave Exclusion

The key innovation is that graves don't count towards caps. When checking if a tribe can accept new nodes:

alive_count = count(n -> !n.is_grave, values(tribe))
if alive_count >= cap
    throw AIMLNodeError("population cap exceeded")

This design prevents "cap lock" — a scenario where accumulated dead nodes block all new growth. When an AIML node dies (marked is_grave = true), it immediately frees cap space. The principle: "graves are memory, not bloat" — dead nodes preserve history without eating resources.

2×2 Steal Mechanics

Within each group, nodes are sorted by strength descending. The top half form stealer pairs and the bottom half form target pairs. Each stealer pair picks a target pair and steals one action from each target. The steal is probabilistic: a stealer copies the highest-weighted action from its target with probability proportional to the target's strength. This means strong targets are more likely to have their actions noticed and stolen — a directional knowledge flow consistent with the v7.1 principle that proven patterns propagate, but now in a structured group context rather than a solo gossip exchange.

Markov Remix

After stealing, each stealer pair's combined actions (original + stolen) are remixed through a Markov chain with MARKOV_CROSSOVER_RATE = 0.35. The Markov chain preserves local sequence structure — adjacent actions in the original action packet are likely to remain adjacent in the remix — while introducing controlled variation. This is fundamentally different from the v7.1 approach of injecting random tokens from the sender's pattern into the receiver's pattern with a flat blend factor. The Markov remix ensures that stolen knowledge is structurally coherent: the remixed action packet is a rearrangement of meaningful subsequences, not a random shuffle.

ChatterSession Lifecycle

A ChatterSession struct tracks the full lifecycle:

  1. Group formation: Sample 8–16 nodes from the alive non-image population. Clone each into a ChatterNodeClone.
  2. Strength sort: Sort clones by strength descending within the group.
  3. 2×2 pairing: Top half = stealers, bottom half = targets. Pair stealer[i] with target[i].
  4. Steal phase: Each stealer copies one action from its paired target (probabilistic, strength-weighted).
  5. Remix phase: Stealer pairs' combined actions are Markov-remixed with crossover rate 0.35.
  6. Graft phase: Remixed actions are grafted onto the original node's action packet (append, no overwrite).
  7. Grave sweep: Nodes that were grav-ed during the session are cleaned from the clone pool.
  8. User input queue: Any user input that arrived during the session is queued via INPUT_QUEUE and drained after session completion.

15.10 Per-Activation Relational Jitter v7.9

Relational match-score components — weight, the +2.0 exact-match contribution, the +1.0 partial-match contribution, the +0.5 orthogonal penalty, and the ×0.1 final dampener — used to be deterministic constants. Every activation produced the same numeric score for the same triples. That made exact ties between competing nodes impossible to break naturally, and forced weaker neighbors to lose every single time to stronger ones.

The RelationalJitter module (src/RelationalJitter.jl) injects a tiny zero-mean nudge into each scored value at activation time. The bullseye is preserved in expectation — over many activations the mean score converges back to the deterministic score — while any single activation sees a slightly perturbed score that can tip exact ties toward a neighbor.

Grug throw rock at target many times. Rock always land EXACT center. Boring! Grug add tiny shake to rock right before it land — rock land slightly off, different direction every throw. Over many throws, average is STILL center. Bullseye survives. But now when two rocks tie, tiny shake break tie — quiet rock sometimes win fair coinflip. More alive.

The Formula

For every scored value x, each call returns:

    jittered = x + δ     where     δ ~ U(-ε·|x|, +ε·|x|)

with ε = JITTER_RATIO (default 0.03). The distribution is symmetric so E[δ] = 0. Therefore E[jittered] = x — the expected value is the bullseye. The variance is Var[jittered] = ε²·x² / 3, bounded and small.

Snap-Back Property

"Snap back to normal" is the statistical LLN (law of large numbers) convergence: a single activation is displaced at most ±ε·|x| from the deterministic value, but the empirical mean over N activations converges to x at rate O(1/√N). Downstream aggregators that average scores (or that run many cycles and pick modes) see the deterministic result; downstream tie-breakers that consume single-activation scores see naturally broken ties.

15.11 AIML Activation Jitter: Strength, Delta, and Coin Gates v7.10

Version 7.10 extends the per-activation entropy framework of §15.10 from the relational dialectics layer into the AIML executive-node layer. Every bullseye value that the AIML system touches during a cycle — the initial strength of a freshly-born node, every reward or penalty delta applied to an existing node, and the three 50/50 coin gates used by record_fire!, apply_aiml_right!, and apply_aiml_wrong! — now runs through RelationalJitter before it takes effect.

Rocks in the top brain used to march in lockstep. All born at exact strength 5. All get exact +1 when right, exact -1 when wrong. All flip fair coin with EXACT 50/50 bias. Too clean! Same long-run fairness, but one rock get slightly bolder, neighbor get slightly quieter, ties break naturally. Top brain breathes instead of ticking.

Two Jitter Shapes, One Primitive Library

The AIML layer uses two jitter primitives from RelationalJitter.jl, each with a distribution shape matched to what is being perturbed:

PrimitiveShapeRangeUsed For
jitter_strength(s) / jitter_delta(d) Proportional uniform x + U(-ε·|x|, +ε·|x|), ε=0.03 Initial strength, reward/penalty deltas (sign preserved, zero passes through)
jitter_coin_threshold(p) Additive uniform, clamped clamp(p + U(-ρ, +ρ), 0.01, 0.99), ρ=0.01 The three rand() < 0.5 AIML coin gates

Why two shapes? For a value like strength = 5.0 the “bullseye” is the value itself, so a proportional nudge scales sensibly: a 3% window means [4.85, 5.15], completely independent of what the strength happens to be. For a probability like p = 0.5 the “bullseye” is near zero under the ratio interpretation — a proportional nudge would be almost nothing — so an additive nudge with a tight absolute window is the right shape. The coin-threshold clamp to [0.01, 0.99] prevents a single freak draw from silently turning a 50/50 gate into an always-yes or always-no gate, which would be a catastrophic behavior shift the no-silent-failure policy demands be impossible.

Preserved Invariants

All hard spec rules from the AIML system (§8.5) continue to hold under jitter:

  • Strength bounds: strength is still clamped to [AIML_STRENGTH_FLOOR, AIML_STRENGTH_CAP] after the jittered delta is applied. Jitter cannot push a node out of the legal range.
  • Grave transition: triggered by strength ≤ AIML_STRENGTH_FLOOR on the clamped new strength. A tiny negative nudge on a near-zero delta cannot cause a spurious grave transition because the trigger condition is checked on the post-clamp value.
  • Zero passes through: jitter_value(0.0) = 0.0 exactly. A caller that requests an exact-zero delta still gets zero, and the is_grave flag on zero-strength survival remains deterministic.
  • Honest-net-loss contract: /aimlWrong applied to a node that gained strength this cycle MUST end strictly below cycle-start strength. Under jitter this still holds: the penalty magnitude passed to _apply_strength_delta! is AIML_STRENGTH_DELTA + prior_gain > 0, and the jittered delta preserves sign because ratio < 1. The jitter applied once inside _apply_strength_delta! — the caller does not double-jitter, so the net-loss guarantee stays tight.
  • Long-run 50/50 behaviour: because the coin-threshold nudge is zero-mean and the rand() draw is independent of the threshold draw, the expected fire rate over a cycle integrates back to 0.5. Empirical rate across 2,000 record_fire! trials measured in the test suite stays within ±0.05 of 0.5.

Values that carry semantic contracts rather than numeric targets are deliberately left alone:

  • Booleans is_grave, gained_this_cycle, fired_this_cycle, voted_this_cycle — no meaningful entropy on flags.
  • The grave marker assignment on strength ≤ FLOOR — reaching zero means grave, categorically.
  • The was_grave_before comparison inside /aimlWrong — boolean state, semantic contract.

15.12 /brainstorm: Scoped Heavy-Jitter Override v7.11

Versions 7.9 and 7.10 established a small, always-on per-activation nudge (3% of value, ±1 percentage-point of probability) that breaks exact ties and prevents sibling lockstep without measurably disturbing long-run semantics. Version 7.11 adds a complementary mechanism for the situation where the engine is clearly stuck — the same top vote wins five cycles in a row, or an AIML tribe has settled into a single-pattern rut — and needs a bigger shove to explore alternatives. The /brainstorm CLI command and its underlying with_brainstorm_jitter(f) scope primitive do exactly that: temporarily widen the jitter window for the duration of one mission, then restore the default ratios bit-exact on exit.

Grug small-nudge rock every throw, rock still hit bullseye. Good for texture, bad when rock always STUCK at same bullseye and other rocks never win. When Grug want to think harder about hard problem: Grug grab different slingshot for ONE throw — slingshot that throw wider. Rock goes farther from bullseye, maybe lands different pile, maybe find new path. Then Grug puts slingshot back. Next throw: normal. No permanent change. Just brainstorm burst then snap back.

The Shape: Simulated-Annealing-Lite

Brainstorm is not true simulated annealing — there's no cooling schedule, no acceptance criterion, no meta-loop that decides when to widen. The primitive is stripped down to the single useful bit: a scoped override. Inside the scope, both the proportional value-jitter ratio and the additive coin-threshold ratio jump to their "far-jump" settings. Outside the scope, everything is exactly as v7.9/v7.10 left it.

SettingDefault (v7.9/v7.10)Brainstorm (v7.11)Hard cap

Coin-threshold ratio0.01 (±1pp)0.05 (±5pp)0.10

Both brainstorm settings stay strictly below the permanent hard caps, so every validator in set_jitter_ratio! and set_jitter_coin_ratio! continues to accept the in-scope values. Sign preservation on deltas still holds (ratio < 1), so the honest-net-loss contract of §15.11 continues to hold inside a brainstorm scope without further guarding.

The API: Lexically Scoped, Exception-Safe

with_brainstorm_jitter(f; ratio, coin_ratio) is the only public entry point. It takes a zero-argument callable f, temporarily replaces the two ratios, invokes f(), and restores the ratios on every exit path. The implementation is the obvious try/finally: 

    function with_brainstorm_jitter(f; ratio = 0.08, coin_ratio = 0.05)
        validate_ratios(ratio, coin_ratio)  # throws before state mutation
        saved_r, saved_c = atomic_push_ratios(ratio, coin_ratio)
        try
            return f()
        finally
            atomic_restore_ratios(saved_r, saved_c)
        end
    end

The critical design points:

  • Validation before mutation. NaN, Inf, negative, or over-cap ratios throw JitterError before any state change. A rejected call leaves the system bit-exact identical to its pre-call state, so the caller can retry without leaked state.
  • Atomic push/restore. Both the ratio read and ratio write happen under _CONFIG_LOCK, so two concurrent tasks cannot both see depth == 0, both push, and have one's restore undo the other's push. The lock serializes scope entry and exit globally.
  • Finally-block restore. Whether f returns normally, throws, or does an early return, the finally block restores the saved ratios. The scope cannot leak.
  • Nested scopes refused loudly. The scope uses a depth counter (_BRAINSTORM_DEPTH); entering when depth is non-zero throws JitterScopeError. The alternative — silently taking the outermost (or innermost) ratios — would be a correctness bug disguised as a convenience, so we refuse it. If nested scopes become useful later, the fix is one line: swap the refusal for a stack of saved ratios.
  • Orthogonal to global enable toggle. Brainstorm controls magnitude; disable_jitter!() controls on/off. Entering a brainstorm scope while jitter is globally disabled leaves every jitter_* primitive an identity function — the raised ratios are present in state but never applied. Scope exit does not re-enable the global toggle.

CLI Surface

The /brainstorm command in src/Main.jl is a thin shell around the scope primitive. It has the same capture shape as /mission (one text blob) and feeds the captured text straight into process_mission, but wraps the call in with_brainstorm_jitter():

The module-level refusal remains the correctness guarantee — the CLI guard is UX.

Where the Entropy Goes

Because every bullseye value in v7.9 (relational dialectics) and v7.10 (AIML strength / delta / coin gates) routes through RelationalJitter, a brainstorm scope automatically widens every one of them without any per-call-site wiring. Specifically, during a brainstorm mission:

  • Each relational match-score contribution (+2.0·w, +1.0·w, +0.5·w, the ×0.1 final dampener) gets ±8% instead of ±3%. Previously-unbreakable ties in the 0.5–1.0 score range now swap winners roughly 15–20% of the time instead of 5%.
  • Each AIML initial-strength seed and every strength delta gets ±8% instead of ±3%. The 50/50 bias in sibling reward/penalty shifts to a noticeably less symmetric distribution per cycle.
  • All three AIML 50/50 coin gates (record_fire!, apply_aiml_right!, apply_aiml_wrong!) use a threshold of 0.5 ± 5pp instead of 0.5 ± 1pp. Long-run 50/50 is still preserved by the zero-mean property, but single-cycle divergence between sibling contributors is substantially larger.

Invariants Still Held

  • Sentinel pass-through: the hard-requirement-miss value (-9999.0) is returned exactly unchanged, regardless of ratio. Verified in the v7.9 test suite; the brainstorm path does not go near the sentinel logic.
  • Sign preservation on deltas: because the brainstorm ratio (0.08) is still less than 1, |δ| < |x| so a positive delta stays positive and a negative delta stays negative. The AIML honest-net-loss contract is preserved.
  • Strength clamp: the post-jitter clamp(strength, FLOOR, CAP) in _apply_strength_delta! still fires, so brainstorm cannot push a node out of its legal range.
  • Coin-threshold interior clamp: jitter_coin_threshold still clamps to [0.01, 0.99] even under brainstorm ratio, so no single freak draw can turn a 50/50 gate into a degenerate always-yes or always-no gate.
  • No concurrent scope corruption: the depth counter plus the config lock prevent two tasks from entering scope at the same time. The first wins; the second throws JitterScopeError.

15.13 Context-Intensity Jitter: Per-Message Relevance Snap-Back v7.12

Before v7.12, Fresh Memory was a hard-coded last 5 unpinned messages slice. That worked for a fresh cave, but after a few /status dumps the context window was dominated by irrelevant diagnostic banners while a hot, topic-adjacent message from 20 turns ago fell out of view. v7.12 replaces the fixed slice with a per-message intensity score that rises toward relevance and gets a tiny zero-mean jitter every cycle — a relevance-biased coinflip sample.

Per-Message Intensity Field

Every ChatMessage now carries an intensity :: Float64 clamped to [CONTEXT_INTENSITY_FLOOR=0.0, CONTEXT_INTENSITY_CAP=3.0] with baseline CONTEXT_INTENSITY_BASELINE=0.5. The field is updated every /mission by refresh_message_intensities!(user_input), which for each message:

  1. Tokenises the user input (_tokenize_for_relevance: lowercase, drop short tokens, strip punctuation).
  2. Computes score_message_relevance (lexical overlap + relational-triple overlap, dynamic triples included so the same verb extractor used by the engine populates relevance).
  3. Snaps back toward relevance: snapped = intensity + CONTEXT_SNAP_ALPHA · (relevance - intensity), with CONTEXT_SNAP_ALPHA ≈ 0.35.
  4. Adds zero-mean jitter via RelationalJitter.jitter_value(snapped) — the same primitive used by the relational dialectics layer, so /brainstorm scope automatically amplifies the context nudge.
  5. Clamps back into [FLOOR, CAP] via clamp_intensity.

Relevance-Biased Fresh Memory Coinflip

When AIML builds its memory block, each unpinned message is coin-flipped newest-first with p = clamp(intensity / CAP, COIN_P_FLOOR, COIN_P_CEIL). A hot message (intensity = 2.0) surfaces with p ≈ 0.67; a cold banner (intensity = 0.1) with p ≈ 0.03. Pinned messages bypass the coinflip entirely. The final Fresh Memory is capped at MAX_FRESH_CONTEXT=8, and the selected message ids are cached in LAST_SELECTED_MSG_IDS so /right and /wrong can reinforce or penalise the context that led to the last answer.

Feedback Loop: /right and /wrong

v7.12 also wires the context layer into the feedback loop. /right bumps intensity of every message that contributed to the last Fresh Memory by CONTEXT_FEEDBACK_RIGHT_DELTA = +0.5; /wrong sags them by -0.5. Clamp always holds. If no prior mission populated LAST_SELECTED_MSG_IDS, feedback is a no-op — no silent failures, no spurious side effects.

Why jitter context? A deterministic top-5-by-intensity would lock in the early relevance hits and starve late-topic messages. The coinflip preserves stochastic exploration in the context layer, exactly mirroring the vote layer.

15.14 Two-Stage Fresh Memory Gate: Threshold + Coinflip v7.13

v7.12's coinflip works perfectly for caves with tens of messages. For caves approaching the MAX_HISTORY=10,000 cap, the coinflip pool itself becomes the bottleneck — flipping 10,000 coins per /mission is both slow and unnecessary. v7.13 front-loads a threshold gate before the coinflip.

Auto-Tuned Intensity Threshold

auto_tune_intensity_threshold(unpinned) binary-searches the intensity threshold that lands the eligible set (messages with intensity > threshold) in the target band [CONTEXT_ELIGIBLE_MIN=5, CONTEXT_ELIGIBLE_MAX=10]:

  • If count > MAX → raise threshold (narrow the pool).
  • If count < MIN → lower threshold (widen the pool).
  • Otherwise → stop, return current threshold.

Search space is [FLOOR, CAP], with budget CONTEXT_THRESHOLD_STEPS=12 (log2 of CAP resolution is plenty since intensities are real Float64s). Edge cases:

  • Fewer than MIN unpinned total → threshold = FLOOR so everything passes (cannot conjure messages).
  • All identical intensity → binary search converges just under that value, all pass (no infinite loop).
  • Jitter drove everyone to FLOOR → returns FLOOR; the downstream coinflip still narrows.

Pipeline Ordering

pinned                    -> always surface (bypass all gates)
unpinned                  -> [auto-tuned threshold gate]
eligible survivors        -> [intensity-biased coinflip, newest-first]
chosen                    -> capped at MAX_FRESH_CONTEXT
                             cached in LAST_SELECTED_MSG_IDS
Fresh Memory block header -> "threshold=X eligible=N" telemetry

A 10k-message cave now puts roughly 5–10 messages through the coinflip per mission, not 10,000. The pool stays tight; stochastic exploration character from v7.12 is preserved; context-pollution blow-up is killed.

15.17 Chunked MESSAGE_HISTORY Scans with DONE Checkpoints v7.17

Every Fresh Memory scan site used to walk MESSAGE_HISTORY in one tight loop. On a 10,000-message cave that loop is an uninterruptible CPU burst — other Tasks (CLI reader, phagy forensics, chatter) stall until the whole scan completes. v7.17 breaks every such scan into CONTEXT_SCAN_CHUNK=1000-sized batches, publishes a DONE checkpoint after each batch, and yields the scheduler before the next batch begins.

The DONE Checkpoint Contract

Every chunked scan site calls a single shared helper after each batch:

15.18 Lobe Topicality Gate + Semantic-Bridge Exception v7.18

Problem (v7.17 and earlier): all lobes contributed to every vote. The user's diagnosis: "lobes with unrelated data should just completely inhibit — don't pick their votes. Other lobes should only fire if there is some semantic relation that is not directly pattern based."

v7.18 installs an engine-level Lobe Topicality Gate immediately before scan_and_expand returns its vote pool. Every lobe's subject string is thesaurus-expanded and intersected with the thesaurus-expanded mission tokens. A lobe passes the gate if

                ≥ LOBE_TOPICALITY_FLOOR   (default 0.15)

The asymmetric min(·,·) denominator is deliberate: a short mission with a single strong keyword should still wake its domain lobe even if the lobe's subject enumerates several adjacent concepts. Using the smaller side avoids dilution in both directions.

Critical contract: raw pattern-scan hits do not auto-unmute a lobe. If a cooking node's pattern happens to lexically match a physics mission, the cooking lobe stays muted — the node must cross a semantic bridge to vote. This matches the user's explicit directive that bridges "not be directly pattern based."

Semantic-Bridge Exception

  1. Dynamic Relational-Triple bridge. The node's relational_patterns contains a verb that also appears in the mission's live relational triples extracted at runtime via extract_dynamic_relational_triples(mission_text, 3). This is the key insight — bridges are evaluated against triples parsed from this specific mission, not just static triples stored on the node at creation time. High-resolution mode captures compound subjects, nested relations, and causal chains.
  2. Required-Relation Verb bridge. The node's required_relations list shares a verb with at least one node living in an eligible (not muted) lobe. A muted-lobe node that requires transforms can still vote if a physics-lobe node also requires transforms.
  3. Bridge (formerly /nodeAttach). The node is bridged — via bridge_nodes! — either to a node in an eligible lobe, or from a node in an eligible lobe. A bridge is an explicit operator-issued cross-lobe relation, so it short-circuits the gate even when no verb or triple matches.

Failure to cross any of these bridges drops the node from the vote pool entirely. The gate runs after all three expansion passes (drop-table, cross-lobe cascade, cascade relay) so muted-lobe nodes introduced by cascade or relay are also filtered.

15.19 Big-Number / Small-Number Coherence (bidirectional cheap-scan fusion) v7.19

Grug scan rock forward AND backward. Get two confidence numbers. Old brain just average them. But average is DUMB. Two tiny numbers that agree look same as two BIG numbers that agree. That is a lie! Big rocks matter more than small rocks. Grug new brain look at BOTH the difference AND how big the rocks are. If both big AND close → REAL match, high coherence. If both tiny AND close → just noise agreeing with noise, LOW coherence. If one big one zero → disagreement, LOW coherence. Also: when two floats very close together, subtraction kills precision. Grug new brain don't care, it normalizes against the biggest rock so tiny float noise can't flip the result.

v7.19 replaces the arithmetic-mean confidence smoother in _bidirectional_cheap_scan with a magnitude-aware coherence fusion. For tier-1 simple patterns (signal length ≤ 3), the scanner runs forward and reversed cheap scans to catch order-flipped matches (“man bites dog” vs “dog bites man”). Before v7.19 the two confidences were fused by naive averaging: smoothed = (forward + reverse) / 2. This had two structural defects:

  • It hid asymmetry. (0.9, 0.1) and (0.5, 0.5) both average to 0.5, but the first is real disagreement and the second is real agreement. The orchestrator downstream could not tell them apart.
  • It was vulnerable to catastrophic cancellation. Two confidences that differ only in the trailing digits of a Float64 representation produced an averaged output whose meaningful bits had already been lost. Noise was indistinguishable from signal.

The replacement is PatternScanner.big_number_small_number_coherence(forward, backward), now exported from the top-level GrugBot420 module. Given two confidence scalars, the fusion is:

abs_fwd        = |forward|
abs_bwd        = |backward|
magnitude_floor = max(abs_fwd, abs_bwd)          # strongest signal either side
magnitude_mean  = (abs_fwd + abs_bwd) / 2        # mean signal strength
absolute_delta  = |forward - backward|

if magnitude_floor < 1e-9:
    return 0.0                                   # both scans saw nothing; silence is not agreement

relative_delta  = absolute_delta / magnitude_floor
agreement       = 1.0 - clamp(relative_delta, 0.0, 1.0)
coherence       = clamp(agreement * magnitude_mean, 0.0, 1.0)

The result is a value in [0.0, 1.0] that semantically encodes four distinct regimes:

  • Both strong and close (agreement ≈ 1, magnitude_mean ≈ high) → high coherence. Real bidirectional match.
  • Both tiny and close (agreement ≈ 1, magnitude_mean ≈ 0) → low coherence. Two scans that saw almost nothing cannot agree on something.
  • Strong but disagreeing (agreement ≈ 0) → low coherence even though the magnitudes are large. Two scans that both fired confidently but on different patterns are not coherent.
  • One strong, one zero → coherence collapses via agreement = 0, correctly rejecting one-sided hits that the pure arithmetic mean would have rewarded.

Catastrophic-cancellation immunity. When forward and backward differ only by sub-ULP floating-point noise, absolute_delta is tiny, relative_delta is tiny, agreement is ≈ 1.0, and the output is essentially magnitude_mean — the noise never drives the result. Subtraction happens once and its output is immediately normalized against a magnitude that dominates the noise floor.

Input hygiene. The function rejects NaN and Inf on either operand with a PatternScanError. NO SILENT FAILURES — poison values never propagate into the vote pool as apparently-valid confidences.

Engine integration. _bidirectional_cheap_scan now ends with:

smoothed_conf = big_number_small_number_coherence(forward_conf, reverse_conf)

Miss contributions on one-sided hits are still computed the same way (threshold - 0.01), but they are fused with the hit via coherence instead of simple average. A partial-reversal hit now produces a moderate coherence rather than a spike or a zero. The PatternNotFoundError propagation when both directions miss is unchanged.

Test coverage — test/test_big_number_small_number_coherence.jl

74 sub-assertions across 8 test groups:

  2. [B] Catastrophic-cancellation immunity: sub-ULP noise between two large near-equal confidences yields coherence ≈ magnitude_mean, not a noise-driven garbage output.
  3. [C] Asymmetry detection: (0.9, 0.1) and (0.5, 0.5) have identical arithmetic mean but coherence separates them by > 0.3.
  4. [D] Negative-confidence symmetry: two equal-magnitude negatives agree, mixed signs register as maximum disagreement.
  5. [E] Zero / sub-epsilon inputs collapse to exactly 0.0.
  6. [F] NaN and Inf rejected with PatternScanError on every combination (6 poison cases).
  7. [G] Output always clamped to [0.0, 1.0] across 11 sample pairs including out-of-range inputs (±2.0).
  8. [H] Monotonicity, commutativity, and reflexivity: closer inputs score higher, same agreement at bigger scale scores higher, bnsnc(a,b) == bnsnc(b,a), and bnsnc(x,x) == |x| for all tested x.

15.20 Per-Node NONJITTER Tag + End-Confidence Snap-Back Jitter v7.20

Most rocks SHOULD wiggle. Wiggle keep brain alive, stop brain from locking into one shape forever. But SOME rocks must NOT wiggle. Anchor rock. Calibration rock. "This word always means this exact thing" rock. Those rocks grug give a tag: NONJITTER. Tag say: leave me alone, I stand still. Everyone else keep breathing.

Problem

Before v7.20 the final fused confidence returned by _bidirectional_cheap_scan was whatever big_number_small_number_coherence produced — deterministic given the same forward and reverse scores. Rigid determinism at the decision boundary creates two pathologies:

  1. Identical re-scans lock in identical votes. Two back-to-back scans of the same input against the same node produce bit-equal confidences, which encourages the downstream voting system to treat them as redundant — losing an opportunity for the organic tie-breaking that biological systems use at the margin.
  2. Decision thresholds become cliffs. A node scoring 0.2999 and a node scoring 0.3001 are treated as categorically different by a threshold of 0.3, even though they are essentially the same measurement. Without micro-variance, there is no gradient across the cliff for the consensus layer to smooth over.

The fix is a bounded snap-back jitter applied to the fused confidence (slight_jitter from the PatternScanner module), which adds uniform noise in [-jitter_magnitude, +jitter_magnitude] clamped to [-1.0, 1.0]. The magnitude is itself a function of how close to 1.0 the confidence already is, so high-confidence scores wiggle LESS than mid-confidence scores — a bullseye stays closer to a bullseye.

Why "snap-back"?

The jitter is bounded and symmetric (uniform, not normal). Unlike a random walk, which drifts, a bounded-uniform perturbation always returns in expectation to the unjittered value. Repeated scans produce a cloud of confidences CENTERED on the true fused coherence — the signal is preserved, only the exact landing point jitters. This is the "snap-back breath" of the system: inhale, exhale, never leave the chest cavity.

The NONJITTER opt-out

Some nodes MUST produce bit-stable confidences. Examples:

  • Canonical-form anchors. A node representing the exact string "error" in a programming-language lobe must score the same every time it encounters "error" — the downstream AIML pipeline may key off that score for routing decisions.
  • Calibration nodes. Nodes used as reference points for threshold tuning would have their calibration curves smeared by even bounded jitter, making the tuning process unstable across runs.
  • Boot-time seed nodes. Nodes created by the initial stochastic-emotion growth that need to survive specimen round-trips with EXACTLY preserved scoring semantics.

These nodes carry the tag "NONJITTER" in their required_relations vector. When the scanner processes such a node, the post-fusion jitter is skipped and the fused coherence is returned as-is.

Design: tag, not field

NONJITTER is implemented as a string in the existing required_relations::Vector{String} field of the Node struct. This was an explicit design choice with concrete payoffs:

  • Zero struct churn. No new field, no serialization migration, no specimen version bump. Existing saved specimens load unchanged, and new specimens carry the tag automatically because required_relations is already persisted.
  • Unified namespace. Relational rules and behavioral flags share the same bag. Future flags (e.g. NOTVOTE, PINNED, HIDDEN) can follow the same pattern without further schema changes.
  • Idempotent API. set_nonjitter!(node) called twice leaves exactly one tag. clear_nonjitter!(node) on an untagged node is a safe no-op, NOT an error — the function's contract is "after this call, tag is absent", which is true either way.
  • Fast check on hot path. is_nonjitter(node) is an O(n) membership check on a typically ≤4-element vector — effectively constant time, no lock, no allocation. Benchmarked at >1M calls/second.

Wiring

In scan_and_expand, the bidirectional cheap-scan call threads the tag through:

_, token_conf = _bidirectional_cheap_scan(
    target_signal, node.signal;
    threshold=0.3,
    nonjitter=is_nonjitter(node)
)

Inside _bidirectional_cheap_scan, after fusion:

smoothed_conf = big_number_small_number_coherence(forward_conf, reverse_conf)
final_conf = nonjitter ? smoothed_conf : slight_jitter(smoothed_conf)

The kwarg defaults to false, preserving backward compatibility — callers that existed before v7.20 continue to get the default (jittered) behavior without code changes.

What NONJITTER does not silence

The tag governs ONLY the post-fusion micro-jitter on the final confidence. It does NOT affect:

  • Per-window jitter inside cheap_scan. That is substrate-level hardware-variance modelling. It runs inside both the forward and the reverse passes regardless of NONJITTER, because the scanner's window-by-window similarity math depends on it for non-degenerate matching behavior.
  • Relational-weight jitter in the RelationalJitter module. That is controlled by the global _JITTER_ENABLED flag and is orthogonal to node-level tags. A NONJITTER node with relational weights will still have those weights jittered by the global system if jitter is enabled.
  • Fusion math itself. big_number_small_number_coherence is deterministic and always runs. NONJITTER silences only the noise layer on top.

15.21 System-Wide NONJITTER Honor v7.21

v7.20 gave grug the NONJITTER tag. But tag only worked in ONE place — post-fusion jitter inside _bidirectional_cheap_scan. If tagged rock went through relational scoring, or cascade relay, or full-lobe scanner, jitter still hit it. That is a lie. If rock has NONJITTER tag, rock should NEVER wiggle, anywhere. v7.21 makes the tag mean what it says: honored EVERYWHERE a node-scoped jitter touches the system.

Problem

The v7.20 NONJITTER implementation gated exactly one call site — slight_jitter on the fused coherence output of _bidirectional_cheap_scan. But GrugBot applies node-scoped jitter in at least three other places:

  1. evaluate_relational_dialectics (engine.jl) — per-triple jitter_weight and jitter_score from RelationalJitter, plus a final orthogonal-penalty dampener jitter.
  2. fire_cascades! relay (engine.jl) — Gaussian synaptic-noise jitter (randn() * RELAY_CONF_JITTER_SIGMA) added on top of each bridged node's pre-baked base confidence at fire time.
  3. FullLobeScanner.gather_candidates! / activate_candidates! (FullLobeScanner.jl) — slight_jitter applied to similarity scores during candidate gathering and activation confidence during the activation pass.

A tagged node was honest about its intent at the cheap-scan layer, then lied about it everywhere else — it still ended up with jittered relational scores, jittered relay confidences, and jittered activation energies. That defeats the purpose of the tag for canonical-form anchors, calibration nodes, and seed nodes.

Fix: gate all three sites, share the tag through one primitive

Two different plumbing strategies, chosen per call site:

Strategy A — function-level flag where the node is already in scope. evaluate_relational_dialectics already receives the node's required_relations vector as an argument (it uses it for the hard-requirement check). So we check once up front — nonjitter = NONJITTER_TAG in required_relations — and pre-select an identity pass-through (jitter_w = nonjitter ? identity : RelationalJitter.jitter_weight) instead of branching inside the hot double-loop. The same trick is used in fire_cascades! where bridge_node_ref is already fetched from NODE_MAP: we query is_nonjitter(bridge_node_ref) and zero out the Gaussian draw if tagged.

Strategy B — id-set plumbing where the scanner doesn't see Node objects. FullLobeScanner is a standalone module that operates on a Dict{String, Vector{Float64}} of features and never touches NODE_MAP. Adding a Node dependency there would entangle subsystems that are intentionally decoupled. Instead, gather_candidates! and activate_candidates! each gain an optional kwarg nonjitter_ids::Set{String} = Set{String}(). The caller supplies the precomputed tag set; the scanner does an O(1) membership check per candidate and skips slight_jitter when the id is in the set. Default empty set = zero overhead for callers who don't opt in = full backward compatibility.

The bridge: collect_nonjitter_ids()

To cheaply fill the kwarg, v7.21 adds the engine-level helper:

function collect_nonjitter_ids()::Set{String}
    ids = Set{String}()
    lock(NODE_LOCK) do
        for (id, node) in NODE_MAP
            if is_nonjitter(node)
                push!(ids, id)
            end
        end
    end
    return ids
end

Orchestrator call site pattern:

nj_ids = collect_nonjitter_ids()
gather_candidates!(scanner, features; nonjitter_ids=nj_ids)
activate_candidates!(scanner, features; nonjitter_ids=nj_ids)

The helper is O(N) over NODE_MAP with a tiny constant (each is_nonjitter check is ≤4-element membership). Call once per mission and reuse the set for the duration of the scan — the NONJITTER tag is stable across a single scan because nothing inside the scan hot path mutates it.

One semantic gotcha: NONJITTER is not a required relation

The hard-requirement check in evaluate_relational_dialectics walks required_relations and returns the -9999.0 sentinel if the user's input doesn't supply a matching relation. Before v7.21 this loop treated EVERY string in the vector as a semantic relation to check against user triples. But "NONJITTER" is a behavioral flag, not a semantic relation — the user will never supply an input triple with relation "NONJITTER", so a naive implementation would turn the tag into an always-fire requirement miss and silently drop every tagged node from voting.

The fix is a one-line guard in the loop: req == NONJITTER_TAG && continue. The tag is skipped for requirement checks, honored for jitter gating, and everyone else (existing semantic relations) is unaffected.

15.22 Strength-Driven Solidification v7.22

Grug rocks that prove themselves stop wiggling. Simple rule: if strength climbs high enough, the rock gets the NONJITTER tag automatically. If strength drops, tag comes off automatically. Rock is always honest about whether it is solid right now. No ceremony, no manual bookkeeping, no frozen-in-amber stale values — just strength says solid, tag says solid.

The rule

One line of policy:

ConditionEffect
node.strength ≥ STRENGTH_SOLIDIFY_THRESHOLD NONJITTER tag applied (node is solidified)
node.strength < STRENGTH_SOLIDIFY_THRESHOLD NONJITTER tag removed (node is jittering)

The threshold constant ships at STRENGTH_SOLIDIFY_THRESHOLD = 9.0, which is 90% of STRENGTH_CAP = 10.0. A node must be at the top 10% of the strength band to qualify as a solid anchor.

Lifecycle

  1. Node starts jittering. Fresh nodes spawn with strength = 1.0 by default. Below threshold, NONJITTER is off, all node-scoped jitter sites (v7.21) produce bounded micro-variance on confidence, relational scores, cascade relay, and full-lobe scanner candidate/activation energy.
  2. Node earns strength via bump_strength! (/right feedback, successful fires, coinflip wins on use). When the bump pushes node.strength across the threshold, the post-hook check_solidify_threshold! automatically applies NONJITTER. The engine logs:
    [ENGINE] 💎 Node $(id) solidified (strength=9.0 ≥ 9.0) — NONJITTER tag applied.
  3. Node answers bit-stable until its strength changes. Because all node-scoped jitter sites honor NONJITTER system-wide (v7.21), a solidified node's confidence and relational scoring are deterministic for the same input.
  4. Node loses strength via penalize_strength! (/wrong feedback, coinflip losses). When the penalty drops node.strength below threshold, the post-hook automatically removes NONJITTER. The engine logs:
    [ENGINE] 💧 Node $(id) softened (strength=7.0 < 9.0) — NONJITTER tag removed, jitter resumed.
  5. Node can re-solidify. If strength climbs back above the threshold later, the tag comes back on. No frozen state carries across — confidence is always computed fresh from the current query against the current pattern. Solidification only suppresses jitter; it does not cache or freeze the computation inputs.

Why no frozen confidence (and an earlier false start)

An earlier sketch of this feature tried to freeze the node's last-computed confidence value at solidify time, replaying that stored value on subsequent scans instead of recomputing. That turned out to be wrong-shaped: confidence is a per-query output — it depends on what the user just typed, matched against the node's pattern. Freezing the confidence would make the solidified node return the same answer for COMPLETELY DIFFERENT queries, which collapses responsiveness.

The right mental model: NONJITTER makes a solid node return the same answer for the same query. It does not make the node return the same answer for every query. Confidence computation remains fully alive; only the micro-jitter noise layer on top of it is suppressed.

Hooks

The v7.22 wiring is intentionally minimal — two single-line additions at the tail of the existing strength-change functions:

function bump_strength!(node::Node)
    if rand() < 0.5
        lock(NODE_LOCK) do
            node.strength = min(node.strength + 1.0, STRENGTH_CAP)
        end
        check_solidify_threshold!(node)   # v7.22 post-hook
    end
end

function penalize_strength!(node::Node)
    if rand() < 0.5
        lock(NODE_LOCK) do
            node.strength = max(node.strength - 1.0, STRENGTH_FLOOR)
            if node.strength ≤ STRENGTH_FLOOR && !node.is_grave
                node.is_grave    = true
                node.grave_reason = "STRENGTH_ZERO"
            end
        end
        check_solidify_threshold!(node)   # v7.22 post-hook
    end
end

check_solidify_threshold!(node) is idempotent — it applies or removes the tag only when the current state disagrees with the strength threshold, so calling it repeatedly (as happens across many bumps in a row) is cheap and never produces duplicate tags or churn.

Contract with manual tagging

Users may still call set_nonjitter!(node) manually to force the NONJITTER tag on a node regardless of strength — useful for calibration or reserved anchor nodes that must never jitter even at low strength. However, the next time check_solidify_threshold! runs (i.e. the next time the node's strength changes), v7.22 will overwrite the manual tag to reflect the strength rule.

If you want a permanently tagged node that is immune to the v7.22 automation, do NOT route strength-change commands at it (e.g. keep it out of the voting population so bump_strength! / penalize_strength! are never called on it). The simpler alternative: keep the node's strength above the threshold by whatever means you prefer, and v7.22 will keep the tag on for you.

Universality

v7.22 is universal across all node types. Every Node carries the strength::Float64 field (image nodes, text nodes, AIML-seeded nodes, attached nodes, boot-time seeds). Every strength change goes through bump_strength! or penalize_strength!. Therefore every node is eligible for auto-solidification on the same rule. No node type is special-cased.

15.23 InputDecomposer: Compound-Query Decomposition v7.23

Grug see one input. User type one line. But user pack THREE question in one breath: "what time is it ALSO what is dinosaur AND what is 2+2". Old grug treat as one big jumble and mash answers into stew. New grug have FRONT-DOOR splitter that look for conjunction + question marker on BOTH sides of the joint. Three sub-subjects, three group IDs, three independent scan passes, ONE coherent answer. "bread and butter" stay one subject because no question on either side. Heuristic, not parser.

Module: src/InputDecomposer.jl (549 lines). Sits at the very front of the pipeline, before scan_and_expand.

Exports: DecomposedSubSubject, decompose_input, is_compound, InputChunk, chunk_boundaries.

Strain Energy & Hippocampal Warrant v7.50

The EphemeralMLP now carries two additional fields that couple the vote-list transformer to the hippocampal modulation system:

When strain exceeds the threshold, the MLP signals that the current vote list is structurally unusual — either highly novel or of unusually low quality relative to the familiar baseline.
  • hippocampal_warrant_active::Bool — a flag indicating whether the HippocampalModulator has issued a warrant for this cycle. When strain exceeds STRAIN_THRESHOLD, the MLP sets hippocampal_warrant_active = true, which causes the hippocampal dispatch to reserve additional action-log steps for the strained cycle. This is the mechanism by which an unusual vote list triggers deeper hippocampal sequencing: the MLP detects the anomaly and the hippocampus allocates more processing budget.

The function get_strain_energy(mlp_state) returns the current strain value, is_hippocampal_warrant_active(mlp_state) returns the warrant flag, and dampen_strain!(mlp_state) decays the strain toward zero between cycles, preventing a single unusual cycle from permanently activating the warrant. Strain computation is integrated into mlp_forward — it is computed from the same hidden-layer activation that drives the sigmoid/ReLU routing decision, so no separate forward pass is needed. PhaseMix in the hidden layer contributes to both the activation routing and the strain computation, making the two outputs coherent by construction.

15.33 TonalJudge: Token Bag + Common-Sense Frame Selection v7.21

TonalJudge sits between the predictor and the scaffold. It does not replace either. It is a structure-preserving translation layer that lifts a PredictionResult into a token bag, judges the situation using common-sense mode selection, and emits a single FrameHint that downstream scaffolds can dispatch on.

Token Alphabet

  • TOK_FORM — question_form, imperative_form, fragment, dangling_chain
  • TOK_INTENSITY — high_arousal, low_arousal, concentrated, diffuse
  • TOK_COHERENCE — coherent, incoherent, neutral_coherence

15.34 InputLedger: Background Thread Mining User Input into Cell Growth Data v7.25

The user input log IS the data. Every word the user types carries latent structure — which topics cluster together, what relational triples emerge, what concepts co-occur. InputLedger watches MESSAGE_HISTORY, consumes ONLY fresh entries (tracked by a hash ledger), and digests them into batches that feed mitosis and relational governance. It is the gut that digests input into nutrients for cell growth.

Hash Ledger

The core deduplication mechanism is a Dict{UInt64, Bool} where the key is a hash of role|text and the value is always true (consumed). If a hash is NOT in the ledger, the entry is fresh. If it IS, the entry has already been digested and is skipped. No input is ever double-processed. The ledger is bounded at LEDGER_CAP = 20000 (2× MESSAGE_HISTORY cap of 10k, accounting for replaced/pinned-evicted entries creating new hashes).

Background Thread

An @async loop runs continuously: walk MESSAGE_HISTORY, skip consumed entries, collect fresh entries into a batch. When length(fresh) ≥ MIN_BATCH_THRESHOLD (50), the batch is processed for co-occurrence data (intensity increment INPUT_CO_OCCUR_INCREMENT = 2.0 — user-typed co-occurrence is a STRONGER signal than scan-cycle co-firing because the user CHOSE to put those concepts together). After processing, entries are marked consumed in the ledger. The thread auto-restarts on crash, logs errors but never dies.

15.37 Anti-Match Nodes: Pattern-Activated Confidence Drain v7.49

Anti-match nodes are a special node class that activates on pattern match but never votes — instead they drain confidence from regular nodes in the same scan pool. They are the system's immune response: when certain patterns should suppress rather than contribute to the response, anti-match nodes ensure those patterns pull confidence away from the active quorum.

Node Declaration

Anti-match nodes are created with is_antimatch_node=true in the create_node call. This flag is stored in the Node struct and checked during the scan cycle. Anti-match nodes participate in pattern matching normally — they appear in the expanded set — but their confidence is tagged with antimatch=true in the vote entry tuple.

Confidence Drain Mechanics

When an anti-match node appears in the expanded set, instead of contributing its confidence to the vote pool, it drains confidence from regular (non-anti-match) nodes in the same lobe. The drain rates are:

NONJITTER anti-match nodes (fixed drain per activation):
drain = ANTIMATCH_DRAIN_FIXED = 0.03

Jittered anti-match nodes (random drain per activation):
drain = rand() × ANTIMATCH_DRAIN_MAX_JITTER = rand() × 0.05

Anti-match nodes have no strength dynamics — they don't gain or lose strength from firing. They don't bump, don't vote, don't get elected. They are pure suppressors.

Use Cases

Anti-match nodes let the architect declare: "when this pattern matches, the response should move AWAY from that topic, not toward it." For example, an anti-match node on "calculate" in a social lobe drains confidence from math-adjacent nodes, ensuring the social response isn't pulled toward computation. This is the dual of the regular vote system — where votes add confidence toward a topic, anti-match drains pull confidence away from competing topics.

Anti-match nodes are the immune system. They don't fight FOR anything — they fight AGAINST. Pattern-activated suppressors that drain confidence from the wrong kind of activation. No strength dynamics. No voting. Just quiet, persistent suppression of what shouldn't be there.

15.40 LobeOrchestrator: Averages-Curve Lobe Selection v7.25

LobeOrchestrator replaces the v7.18 "topicality mute gate" with an averages-curve lobe scoring system. No lobe is muted up-front. Every lobe collects its node-vote confidences, and the lobe with the highest score wins the orchestration floor. Runner-up lobes can also fire if they meet the pass-through threshold. This produces natural multi-lobe responses without the hard mute/demote cycle of the old system.

Lobe Score Formula

base_avg = mean(all vote confidences in lobe)
top_avg = mean(top-K confidences in lobe) where K = TOP_K_FRACTION × total votes
lobe_score = sqrt(base_avg) × top_avg²

Top averages have exponentially more weight than base averages. A lobe with a few very strong votes outperforms a lobe with many mediocre votes.

Multi-Lobe Pass-Through

A runner-up lobe fires alongside the winner IF:

  1. lobe_score ≥ MIN_PASS_THROUGH_SCORE (0.10), AND
  2. it has ≥ MIN_WINNING_VOTES_PER_LOBE (2) hard-selected votes (small votes piling up to a high average do NOT count).

Equal scores are broken by 50/50 coinflip. Winner goes first, both still fire.

Hard Fire Batch Cap

Each lobe fires in batches capped at HARD_FIRE_BATCH_CAP = 1000 rows. Winner fires first. Runners-up wait their turn and also fire in 1k-capped batches. Cross-talk, bridge, and cascade activations from any of these lobes MUST WAIT until the lobe(s) finish, then fire as their own 1k-capped batch. The FireCounter shared across the entire scan cycle enforces the global ACTIVE_FIRE_CAP = 1000.

Default Lobe Demotion

When any named lobe has hard votes ≥ 1, the default lobe's score is multiplied by DEFAULT_LOBE_DEMOTION_FACTOR = 0.5. This prevents greedy boot patterns in the default lobe from stealing wins from named lobes on inputs the named lobes obviously own. The default lobe can still win if no named lobe has any hard votes, or if its score is more than 2× the strongest named lobe.

15.44 Concurrency Model: Lock Ordering, Snapshots, and the Fire Cap v8.0

The GRUG engine runs multiple concurrent tasks: the main scan cycle, background ledger threads (InputLedger, ChatterResiduals), auto-crystallize sweeps, and the chatter/residual processing loops. With ACTIVE_FIRE_CAP = 1000 and bidirectional bridge mutations happening during scan cycles, deadlock prevention and lock ordering discipline are critical.

Lock Ordering Convention

The system enforces a strict lock acquisition order to prevent deadlocks:

NODE_LOCK → BRIDGE_LOCK (always this order, never reversed)

CO_ACC_LOCK is standalone — never acquired while holding NODE_LOCK or BRIDGE_LOCK

_LEDGER_LOCK (InputLedger/ChatterResiduals) is standalone — never nested with engine locks

Any code that needs both NODE_LOCK and BRIDGE_LOCK must acquire NODE_LOCK first, then BRIDGE_LOCK. The reverse order causes deadlock: if Task A holds BRIDGE_LOCK waiting for NODE_LOCK, and Task B holds NODE_LOCK waiting for BRIDGE_LOCK, both wait forever. The convention is enforced by code review and validated by the 3-step snapshot approach.

3-Step Snapshot Approach

When a function needs data protected by two different locks (e.g., bridge data under BRIDGE_LOCK and node data under NODE_LOCK), it uses the 3-step snapshot approach instead of nested lock acquisition:

  1. Snapshot under Lock A: Acquire Lock A, copy the needed data into local variables, release Lock A. The copy is an immutable snapshot — no concurrent modification can affect it.
  2. Read under Lock B: Acquire Lock B, read the needed data using the snapshot from step 1 as context, release Lock B.
  3. Apply mutations under Lock A: Re-acquire Lock A, apply the mutations computed in step 2, release Lock A. Verify the data hasn't changed since step 1 (the entry still exists, the bridge still connects the same nodes).

This pattern is used by get_bridge_summary(), auto_crystalize_sweep!(), and fire_cascades! (which snapshots the bridge vector under BRIDGE_LOCK before iterating under NODE_LOCK). The cost is an extra lock acquisition, but the benefit is zero deadlock risk.

FireCounter: Shared Active Cap

The FireCounter struct in VoteOrchestrator provides a thread-safe shared counter for the active fire cap. Each scan cycle creates a FireCounter(cycle_id, ACTIVE_FIRE_CAP=1000). Every node fire — primary scan, cascade handoff, bridge relay — increments the same counter. When fire_cap_reached(fc) returns true, ALL fires stop for that cycle. This is the biological attention bottleneck: the engine cannot process more than 1000 active nodes per scan cycle, regardless of how many bridges, lobes, or attachments are competing for attention.

CO_ACC_LOCK Isolation

The RelationalGovernance co-activation accumulator lock (CO_ACC_LOCK) is intentionally isolated from the NODE_LOCK/BRIDGE_LOCK hierarchy. The observe_co_firing! function computes token overlaps outside CO_ACC_LOCK (avoiding NODE_LOCK nesting), then mutates CO_ACC under CO_ACC_LOCK only. This isolation means co-activation observation never competes with bridge creation or node mutation for lock ordering.

Lock ordering is the law. NODE_LOCK before BRIDGE_LOCK, always. No exceptions. The snapshot approach is how you follow the law when you need data from both locks — copy under one, read under the other, apply under the first. Extra lock acquisition, zero deadlock risk. The fire cap is the population control: 1000 nodes per cycle, shared counter, no exceptions. The CO_ACC lock is the isolated cell block — it never touches the main locks, so it can't deadlock against them. Discipline keeps the system alive.

16.5 CascadeBridge: Bidirectional Match-Cascade Handoff System v8.0

The CascadeBridge system replaces the v7.3 one-directional AttachedNode/ATTACHMENT_MAP architecture with a bidirectional match-cascade handoff. Where the old system used a fabricated connector pattern as a middleman between source and target, the new system hands off the unmatched tail of the scanner input — the tokens the primary lobe could not match — directly to the partner lobe's scanner. This is the natural cross-lobe bridge: the match boundary IS the seam, and the seam IS the payload. No connector pattern invention. No middleman. When node A fires, its bridge partners in BRIDGE_MAP[A] receive the unmatched tail. When node B fires, its partners in BRIDGE_MAP[B] receive the unmatched tail from B's perspective. Both directions work. No more one-way derp.

CascadeBridge Struct

mutable struct CascadeBridge
  partner_id::String — ID of the node on the other side of this bridge entry
  seam_tokens::Vector{String} — tokens at the match boundary; populated at bridge time or dynamically at fire time
  base_confidence::Float64 — JIT-baked symmetric pattern overlap + partner strength bonus
  source_lobe::String — lobe ID of the node that created this entry (provenance)
  is_crystalized::Bool — if true, high-confidence votes skip coinflip (deterministic pass-through); low-confidence votes still flip
  crystal_origin::Symbol — :user (manual /crystalize), :auto (semantic-truth triggered), or :none

BRIDGE_MAP: Bidirectional Index

BRIDGE_MAP is a Dict{String, Vector{CascadeBridge}}. When nodes A and B are bridged, both BRIDGE_MAP[A] and BRIDGE_MAP[B] receive an entry. The A→B entry's partner_id is B and source_lobe is B's lobe. The B→A entry's partner_id is A and source_lobe is A's lobe. This bidirectional indexing means fire_cascades! only needs to look up one key to find all partners. Old aliases ATTACHMENT_MAP = BRIDGE_MAP and ATTACHMENT_LOCK = BRIDGE_LOCK preserve backward compatibility throughout the codebase.

JIT Confidence Baking (Bidirectional)

Confidence is computed once at bridge time under NODE_LOCK, then stored in each direction's CascadeBridge. Because the bridge is bidirectional, each direction gets its own independently baked confidence reflecting the partner's strength:

A→B direction:
jit_conf_a_to_b = token_overlap(pattern_a, pattern_b) + (strength_b / STRENGTH_CAP) × 0.5

B→A direction:
jit_conf_b_to_a = token_overlap(pattern_a, pattern_b) + (strength_a / STRENGTH_CAP) × 0.5

At fire time (same formula for both directions):
confidence = max(0.1, base_confidence + jitter_allowed ? randn() × RELAY_CONF_JITTER_SIGMA : 0.0)

bridge_nodes! — Bidirectional Bridge Creation

bridge_nodes!(node_a, node_b; seam_tokens=String[]) creates a bidirectional bridge. Validation is exhaustive and error-first: both nodes must exist in NODE_MAP, neither can be grave, node_a ≠ node_b (no self-bridging), neither side can already have MAX_BRIDGES bridges, and no duplicate bridge can exist in either direction. Under BRIDGE_LOCK, two CascadeBridge entries are created — one for each direction — and pushed into the respective BRIDGE_MAP vectors. Cross-lobe bridges are tagged with a [CROSS-LOBE: A ↔ B] diagnostic. The old attach_node! function is a backward-compat wrapper that splits the pattern into seam tokens and calls bridge_nodes!.

fire_cascades! — Match-Boundary Handoff

fire_cascades!(source_id, active_count, active_cap; unmatched_tail=String[]) replaces the old fire_attachments! with match-cascade handoff semantics. The critical innovation is the unmatched_tail parameter: when the source lobe's scanner matches node A but cannot match certain tokens, those leftover tokens are the bridge payload. They carry provenance — which lobe, which node, what context — so the partner lobe's scanner knows exactly where the handoff came from.

The firing pipeline for each bridge entry is:

  1. Snapshot bridges under BRIDGE_LOCK (copy the vector to avoid concurrent modification races), then release BRIDGE_LOCK.
  2. Active cap gate: current_active ≥ active_cap — biological attention bottleneck. Hard cap shared across all nodes in the scan cycle via FireCounter (ACTIVE_FIRE_CAP = 1000).
  3. Alive check: partner must exist in NODE_MAP and not be grave. Stale bridges (partner deleted/graved) are silently skipped.
  4. Confidence-biased bridge coinflip: confidence_biased_bridge_coinflip(partner_ref, br.base_confidence) biases the relay coinflip by both partner strength and the bridge's base_confidence relative to AIML_CONFIDENCE_THRESHOLD. High-confidence bridges fire more reliably. Weak ones still get a floor chance (20%). Crystalized bridges with confidence ≥ AIML_CONFIDENCE_THRESHOLD skip the coinflip entirely — deterministic pass-through for high-confidence votes. Crystalized bridges with low confidence still flip — confidence must be earned, even for crystal bonds. Non-crystalized bridges always flip with the confidence-biased formula.
  5. JIT confidence with jitter: max(0.1, base_confidence + jitter). NONJITTER nodes get zero jitter. Floor of 0.1 ensures bridged nodes always have some voice in the quorum.
  6. Seam selection: if unmatched_tail is non-empty (live handoff from the scanner), it replaces the bridge's stored seam_tokens. Otherwise the stored tokens are used (backward compat for manual bridges).
  7. Cross-lobe logging: cross-lobe handoffs emit diagnostic lines with provenance (source lobe, partner lobe, seam text, confidence).

Returns Vector{Tuple{String, Float64, String}} — partner_id, confidence, seam_text triples for the generative pipeline. The old fire_attachments! is a backward-compat wrapper that calls fire_cascades! without unmatched_tail.

Crystalize System

crystalize_bridge!(node_a, node_b; origin=:user) marks a bridge as crystalized on both sides (bidirectional). Origin must be :user (manual) or :auto (automated). Crystalized bridges with high-confidence votes (≥ AIML_CONFIDENCE_THRESHOLD) skip the coinflip and fire deterministically. Crystalized bridges with low-confidence votes still flip — confidence must be earned, even for crystal bonds. The crystal_origin field tracks provenance — manual marks are never revoked by the auto-sweep.

auto_crystalize_sweep!() walks every bridge in BRIDGE_MAP and auto-crystallizes bridges where both the source AND partner nodes have strength ≥ CRYSTAL_AUTO_STRENGTH_FLOOR and semantic truth score ≥ CRYSTAL_AUTO_SEMANTIC_FLOOR. Both sides must be strong — it takes two proven nodes to form a crystal bond. One strong side is not enough; this prevents weak nodes from free-riding on a strong partner's crystal status. The semantic truth score (_semantic_truth_score) measures how well a node's relational triples are anchored: fraction of triples with registered relation-class verbs, plus bonus for required_relations and non-empty relation_weights. Auto-crystallized bridges are de-crystallized if either side's strength drops below CRYSTAL_AUTO_REVOKE_FLOOR — crystal is a mutual commitment that both sides must maintain. The sweep uses a 3-step snapshot approach to avoid nested lock acquisition (see §15.44 Concurrency Model).

get_bridge_summary — Snapshot-Based Diagnostics

get_bridge_summary() returns a human-readable summary of all bridges. It uses the 3-step snapshot approach: snapshot bridge data under BRIDGE_LOCK, read partner node status under NODE_LOCK, format output without holding any lock. The summary shows bidirectional pairs with partner status, crystalize state (origin tag), source lobe, base confidence, and seam token preview.

Specimen Persistence

Bridges are serialized under the "bridges" key in the specimen save format (replacing the old "attachments" key). Each entry stores node_id, partner_id, seam_tokens, base_confidence, source_lobe, is_crystalized, and crystal_origin. On load, if base_confidence is missing (backward compatibility with older specimens), it is re-computed from token overlap + strength bonus. If seam_tokens is missing, it defaults to an empty vector (populated dynamically at fire time). Old specimen files with "attachments" keys are automatically migrated to the new bridge format.

Design Rationale

The CascadeBridge rework solves three problems with the old AttachedNode architecture. First, the old system was one-directional: when A was attached to B, only A→B relay worked. B never got a chance to relay back to A. This meant every cross-lobe link required two manual attach commands, and the system had no natural reciprocity. Second, the connector pattern was a fabricated middleman — a user-specified string that sat between source and target but added no information. The match-boundary handoff replaces this fabrication with the actual unmatched scanner tokens, which carry genuine provenance from the primary scan. Third, cross-lobe suppression was a hack: the old system suppressed cross-lobe attachments and redirected them to cascade Pass 2, which meant user-defined relay circuitry was silently overridden by the engine's automatic cascade. The new system lets cross-lobe bridges work naturally — the handoff carries lobe provenance so the partner's scanner knows exactly where the signal came from. The biological parallel shifts from pre-wired axonal pathways to commissural fibers: bilateral brain structures that carry signals in both directions, with the seam tokens encoding which signals crossed the midline.

17.5 Signal Presence & Instant Activation v7

The most important performance insight in GrugBot is not about throughput or parameter count. It is about the nature of the signal itself. In transmission-based architectures, a signal races from input to output through a series of discrete steps: tokenize → embed → attend → decode. Latency is the sum of those transmission costs. GrugBot operates differently: the signal does not race because it is already present.

Always-On Ambient Field

The node population is never fully off. Between interactions the system runs at ambient low flow: ChatterMode gossip, PhagyMode maintenance, Hopfield cache warm. When input arrives and intensity rises, nothing is assembled from scratch. The activation is not a signal traveling through the network — it is a state change in a field that was already present and already aligned. Intensity rise is therefore instant: there is no cold-start propagation delay because there is no propagation.

Transmission latency model (conventional):

Ltotal = Ltokenize + Lembed + Lattend + Ldecode

Resolution latency model (GrugBot):

Ltotal = Lcollapse = time for pre-loaded field to reach quorum agreement

When Hopfield cache hits: Ltotal ≈ O(1) — cache lookup only, no scan

When quorum is sure (nodes within 0.05 of peak confidence): fast collapse

When quorum is unsure: full stochastic scan required, collapse is slower

The Inverse Mode: Low-Flow Activation

The same principle operates in reverse. At low ambient flow — when no input is present — the system does not go quiet. It activates in a different mode: ChatterMode. Ephemeral clones of 50–500 nodes exchange patterns on a coinflip (1000+ node population required). No external input is required. The activation is driven by the internal field state. The network restructures itself along lines of internal coherence — consolidating associations, cross-linking related patterns, pruning irrelevant connections. The low-flow state is not a deficit. It is the system maintaining the pre-alignment that makes instant activation possible.

The Entanglement Principle

Quantum entanglement is not faster-than-light communication. Two entangled particles do not exchange a signal when one is measured. The correlation was established at the moment of entanglement. The measurement does not send anything — it reveals a pre-existing relational state.

GrugBot's always-on field operates on the same structural principle. When input arrives and a node fires at high confidence, it is not because the signal raced through the network and found a match. It is because the node was already aligned with that pattern domain. By the time the strength-biased coinflip fires and the node is scanned, every factor determining the outcome has already been established:

  • Pattern vector pre-loaded at node creation
  • Strength pre-accumulated through prior interactions and /wrong feedback
  • Lobe membership pre-established through domain assignment
  • Hopfield cache pre-recorded if this input hash has been seen before
  • ActionTonePredictor pre-shaped the scan field before the first node was reached
  • EyeSystem arousal pre-set the attention gate width
  • NegativeThesaurus pre-stripped inhibited tokens from the input

The scan is not a computation — it is a measurement of a pre-existing state. The response speed is therefore not a function of how fast the computation runs. It is a function of how well the field was pre-aligned.

Formal statement of the resolution latency principle:

Let F(t) be the field state at time t (node strengths, Hopfield cache, lobe topology, arousal baseline). Let I be an input. The response R is determined entirely by F(t) and I. The latency L is the time required for the quorum defined by F(t) to collapse under I. When F(t) is well-aligned to I (high-confidence sure quorum, Hopfield hit), L → 0. When F(t) is poorly aligned (weak unsure quorum, no cache), L is bounded by the full stochastic scan cost.

The field is always loaded. The only question is how well it is aligned to the current input.

Quorum Settlement as Collapse Speed

The sure/unsure quorum split is not only an uncertainty representation — it is a direct measure of field alignment quality. A sure quorum that settles fast (multiple nodes landing within 0.05 of the same peak confidence) means the field was well-aligned to this input. An unsure quorum that requires the full coinflip population means the field was only partially aligned. The collapse took longer not because the system is slower, but because pre-existing alignment was weaker.

Sure quorum condition:

sure = { v ∈ votes : v.confidence ≥ max_conf - 0.05 }

Alignment quality metric:

alignment_quality = |sure| / |votes_total|

High ratio → field well-aligned to input → fast resolution
Low ratio → field weakly aligned → slow resolution, unsure quorum dominant

The same principle governs the Pocket Antikythera hardware architecture. A mechanical system with pre-loaded tension — spring-loaded gear trains, pre-stressed couplings within their tolerance bands — does not transmit force across a distance when triggered. The energy is already present in the mechanical field. The trigger releases a pre-existing alignment. The response is instant not because the trigger is fast but because the energy was already there, the gears already meshed, the slack already calibrated.

Hall effect sensors, optical sensors, and strain gauges in such a system do not detect an incoming signal. They detect the current state of a pre-existing mechanical field. The quorum alignments in the digital feedback layer do not aggregate transmitted votes — they measure the current agreement state of a field that was already running.

Software and hardware, the principle is identical: pre-alignment eliminates transmission. The signal is not in the wire. Only the computation is. And the computation was already running.

v8.0 — CascadeBridge, Bidirectional Bridges, ChatterMode v8.0, Auto-Attach Disabled

  • CascadeBridge bidirectional node-bridge system (src/engine.jl). Replaces the unidirectional AttachedNode / ATTACHMENT_MAP system with CascadeBridge structs stored in BRIDGE_MAP::Dict{String, Vector{CascadeBridge}}. Every bridge is bidirectional: both A→B and B→A receive entries. Fields: partner_id, seam_tokens::Vector{String}, base_confidence, source_lobe, is_crystalized, crystal_origin. JIT confidence baking computes the geometric mean of both partners' last confidences, floored at 0.1. BRIDGE_LOCK::ReentrantLock guards all mutations; lock ordering NODE_LOCK → BRIDGE_LOCK (never reversed) prevents deadlocks. 3-step snapshot approach: snapshot under Lock A, read under Lock B, apply mutations under Lock A — avoids nested lock acquisition entirely.
  • fire_cascades! replaces fire_attachments!. Match-boundary handoff: the unmatched_tail parameter passes unmatched tokens from the activating node's scan to bridged partners, enabling cascade across pattern boundaries. Confidence relay uses pre-baked base_confidence with optional jitter. Deduplication via fired_set::Set{String}. Hard cap: ACTIVE_FIRE_CAP = 1000 via shared FireCounter in VoteOrchestrator.
    v8.0 confidence-biased coinflip: Bridge relay coinflips now use confidence_biased_bridge_coinflip(node, bridge_confidence), which biases by both partner strength and the bridge's base_confidence relative to AIML_CONFIDENCE_THRESHOLD. This replaces the old strength_biased_scan_coinflip (strength-only). Crystalized bridges with confidence ≥ AIML_CONFIDENCE_THRESHOLD skip the coinflip entirely (deterministic pass-through); crystalized bridges with low confidence still flip.
  • Crystalize system. Bridges can be “crystalized” — locked against accidental deletion during specimen save/load cycles. auto_crystalize_sweep! requires both source AND partner nodes to exceed strength and semantic thresholds for auto-crystallization — one strong side is not enough. v7.55/v7.56 — Relation Sigils

  • :relation class sigils in SigilRegistry. Sigils of class :relation expand to the set of alternative relation names for a given canonical relation at bind-time. For example, &rel=causes matches any of causes, leads_to, produces, generates in a node's relational patterns. This dramatically reduces the combinatorial surface of relation-variant nodes: one sigil-bearing node replaces N synonym-specific nodes.   • Strain energy field. EphemeralMLP gains strain_energy::Float64, a scalar tracking accumulated structural stress from repeated weight perturbations. Updated every register_right_feedback! / register_wrong_feedback! call. v7.49 — Anti-Match Nodes

  • Anti-match node flag. Node.is_antimatch_node::Bool field added. When true, the node activates on pattern match but never votes and drains confidence from the primary scan pool. Drain amount: ANTIMATCH_DRAIN_FIXED + rand() * ANTIMATCH_DRAIN_MAX_JITTER per matched antimatch node, where ANTIMATCH_DRAIN_FIXED = 0.03 and ANTIMATCH_DRAIN_MAX_JITTER = 0.05. The jitter prevents deterministic drain that could be gamed.
  • Scan integration. In scan_and_expand, after the primary scan produces confidence scores, any matched antimatch node reduces the top candidate's confidence by the drain amount (floored at 0.0). The antimatch node itself does not enter the vote pool. This provides a clean mechanism for “negative pattern” nodes that suppress specific responses without being selected themselves.

v7.19 — Big-Number / Small-Number Coherence (bidirectional cheap-scan fusion)

  • Replaces naive averaging in _bidirectional_cheap_scan. Tier-1 cheap scans run forward and reversed passes to catch order-flipped matches (“man bites dog” vs “dog bites man”). Before v7.19 the two confidences were fused by arithmetic mean, which (a) could not distinguish real agreement on strong signal from agreement on noise, and (b) was vulnerable to catastrophic floating-point cancellation on near-equal values.
  • New exported function PatternScanner.big_number_small_number_coherence(forward, backward) re-exported at the top level as GrugBot420.big_number_small_number_coherence. Formula: agreement = 1 - |forward - backward| / max(|forward|, |backward|), coherence = clamp(agreement × mean_magnitude, 0, 1). If max(|forward|, |backward|) < 1e-9 (both scans saw nothing), returns 0.0 — silence is not agreement.
  • Four regimes correctly separated: both strong and close → high coherence; both tiny and close → low coherence (agreement on noise); strong but disagreeing → low coherence; one strong, one zero → coherence collapses via agreement = 0.
  • Catastrophic-cancellation immune. When the two confidences differ only by sub-ULP Float64 noise, the subtraction's garbage output is normalized against magnitude_floor, which dominates the noise floor — so the noise never drives the result.
  • NO SILENT FAILURES. NaN and Inf on either operand are rejected with PatternScanError; poison values never propagate into the vote pool as apparently-valid confidences.
  • Test suite: new file test/test_big_number_small_number_coherence.jl with 8 test groups and 74 sub-assertions covering basic behavior, catastrophic- cancellation immunity, coherence ≠ arithmetic mean on asymmetric inputs, negative-confidence symmetry, zero/sub-epsilon collapse, NaN/Inf rejection, output range guarantees, and monotonicity/commutativity/reflexivity.
